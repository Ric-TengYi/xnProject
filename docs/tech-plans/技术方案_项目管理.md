# 技术方案 - 项目管理模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`项目管理`
- 本次覆盖的未实现功能共 `10` 个：
  - 交款数据
  - 处置证清单
  - 项目日报
  - 打卡配置
  - 位置判断配置
  - 出土预扣值
  - 线路配置
  - 违法清单
  - 报表统计
  - 违规配置

当前仓库现状：

- 后端只有 `ProjectsController`，`GET /api/projects`、`GET /api/projects/{id}` 和 `POST /api/projects` 都是空实现；`biz_project` 也只有 `name/code/address/status/org_id` 这类基础字段。
- 前端已有 `ProjectsManagement.tsx`、`ProjectDetail.tsx`、`ProjectsPayments.tsx`、`ProjectsPermits.tsx`、`ProjectsDailyReport.tsx` 页面骨架，当前全部依赖 mock 数据。
- 项目模块同时依赖处置证、打卡、线路、违规、报表多个域，建议先补主档与配置，再补台账与统计。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_project` | 项目主表，补齐租户、编号、单位、总方量、已消纳量、状态、有效期等字段 |
| `biz_project_payment_record` | 项目交款/保证金/冲销记录 |
| `biz_project_daily_report` | 项目日报快照 |
| `biz_project_checkin_rule` | 项目打卡配置 |
| `biz_project_geo_rule` | 位置判断配置与围栏参数 |
| `biz_project_deduction_rule` | 出土预扣值配置 |
| `biz_project_route` | 项目线路主表 |
| `biz_project_route_point` | 线路坐标点和电子围栏点位 |
| `biz_project_violation_rule` | 项目级违规配置 |
| `biz_project_violation` | 项目违法/违规清单 |
| `biz_project_report_snapshot` | 项目统计快照 |
| `biz_project_permit_rel` | 项目与处置证关联关系 |

### 2.2 路由与接口分组

- `/api/projects`
- `/api/projects/payments`
- `/api/projects/permits`
- `/api/projects/daily-reports`
- `/api/projects/checkin-rules`
- `/api/projects/location-rules`
- `/api/projects/deduction-rules`
- `/api/projects/routes`
- `/api/projects/violations`
- `/api/projects/violation-rules`
- `/api/projects/reports/*`

### 2.3 实施顺序

1. 升级 `biz_project`，补齐项目主档字段与项目详情接口。
2. 打通 `打卡配置 / 位置判断配置 / 出土预扣值 / 线路配置 / 违规配置` 这 5 个配置类能力。
3. 再实现 `交款数据 / 处置证清单 / 违法清单 / 项目日报` 这些项目台账页面。
4. 最后统一补 `报表统计` 和导出能力。

## 3. 分功能技术方案

### 3.1 交款数据

#### 1. 数据库设计

- 新增 `biz_project_payment_record`，核心字段建议为 `id/tenant_id/project_id/payment_no/payment_type/amount/payment_date/voucher_no/status/source_type/source_id/remark`。
- 索引建议：`uk_payment_no(tenant_id, payment_no)`、`idx_project_payment_date(tenant_id, project_id, payment_date)`。

#### 2. 接口定义

- `GET /api/projects/payments`：按项目、时间、类型、状态分页查询。
- `POST /api/projects/{id}/payments`：新增项目交款。
- `POST /api/projects/payments/{id}/cancel`：冲销错误台账，并返回最新项目资金汇总。

#### 3. 业务逻辑

- 项目交款台账默认兼容“人工录入 + 合同回写 + 冲销”三类来源，避免后面多套口径。
- 新增或冲销交款时同步回写项目累计交款、剩余额度和最近交款时间。

#### 4. 前端实现要点

- 复用 `ProjectsPayments.tsx` 作为交款台账页，顶部统计卡与列表都改走真实接口。
- 支持从项目详情页带参跳转并默认锁定 `projectId`。

### 3.2 处置证清单

#### 1. 数据库设计

- 复用处置证主表 `biz_disposal_permit`，新增 `biz_project_permit_rel` 存项目与处置证的生效关系。
- 关联表需保留 `project_id/permit_id/site_id/vehicle_id/bind_status/effective_time/expire_time`。

#### 2. 接口定义

- `GET /api/projects/{id}/permits`：查询项目下处置证列表。
- `POST /api/projects/{id}/permits/link`：绑定处置证。
- `DELETE /api/projects/{id}/permits/{relId}`：解除关联。

#### 3. 业务逻辑

- 只允许关联有效、未作废且租户一致的处置证；过期证件自动标红。
- 项目侧清单只展示当前生效关系，历史解绑记录保留在关联表里。

#### 4. 前端实现要点

- 复用 `ProjectsPermits.tsx`，把证件详情抽屉、筛选条件和状态标签切到真实数据。
- 项目详情页增加“处置证”页签，和处置证模块共用详情接口。

### 3.3 项目日报

#### 1. 数据库设计

- 新增 `biz_project_daily_report`，字段建议包含 `tenant_id/project_id/report_date/vehicle_count/trip_count/daily_volume/total_volume/abnormal_count/generate_status/source_hash`。
- 索引建议：`uk_project_report_date(tenant_id, project_id, report_date)`。

#### 2. 接口定义

- `POST /api/projects/daily-reports/generate`：按日期生成或重算日报。
- `GET /api/projects/daily-reports`：按日期和项目分页查询。
- `POST /api/projects/daily-reports/export`：导出日报。

#### 3. 业务逻辑

- 日报优先从消纳记录、打卡记录和违规记录实时汇总，生成后落快照，保证页面秒开。
- 同一天允许重算，但要保留最近一次生成时间和生成人。

#### 4. 前端实现要点

- 复用 `ProjectsDailyReport.tsx`，把“生成报表”和“导出 Excel”按钮改成真实调用。
- 支持单项目明细抽屉，点击项目名称查看当天车辆、趟次和异常明细。

### 3.4 打卡配置

#### 1. 数据库设计

- 新增 `biz_project_checkin_rule`，字段建议为 `project_id/checkin_mode/start_window/end_window/required_points/tolerance_minutes/enable_photo/enable_gps/status`。
- 对 `project_id` 建唯一索引，保证每项目只维护一套生效规则。

#### 2. 接口定义

- `GET /api/projects/{id}/checkin-rules/current`：读取当前打卡规则。
- `PUT /api/projects/{id}/checkin-rules/current`：保存规则。
- `POST /api/projects/{id}/checkin-rules/preview`：模拟判断配置是否可用。

#### 3. 业务逻辑

- 规则至少覆盖“入场、离场、班前、班后”四个场景，并允许不同角色差异化要求。
- 配置变更采用版本号，保证历史打卡记录仍能按旧规则回放。

#### 4. 前端实现要点

- 在 `ProjectDetail.tsx` 配置页增加打卡规则表单和模拟校验按钮。
- 若后续有小程序端，表单字段要与移动端校验参数一一对应。

### 3.5 位置判断配置

#### 1. 数据库设计

- 新增 `biz_project_geo_rule`，保存 `project_id/judge_mode/site_polygon/route_buffer_meters/checkin_radius_meters/stay_threshold_seconds`。
- 若需要地图复用，可拆 `biz_geo_polygon` 存标准 GeoJSON，多模块共享。

#### 2. 接口定义

- `GET /api/projects/{id}/location-rules`。
- `PUT /api/projects/{id}/location-rules`。
- `POST /api/projects/{id}/location-rules/preview`：上传坐标点预演命中结果。

#### 3. 业务逻辑

- 位置判断应服务打卡判定、偏航判定和围栏预警三条链，避免每条链各算一遍。
- 预演接口返回“命中项目/命中围栏/最近线路距离”，便于业务验证配置。

#### 4. 前端实现要点

- 项目详情页增加地图绘制与点位预览区域。
- 配置保存后前端直接展示版本号和最近发布时间，减少误改。

### 3.6 出土预扣值

#### 1. 数据库设计

- 新增 `biz_project_deduction_rule`，字段建议为 `project_id/deduction_mode/daily_quota/reserved_volume/reserved_amount/warning_ratio/effective_date`。
- 索引建议：`idx_project_deduction_status(tenant_id, project_id, status)`。

#### 2. 接口定义

- `GET /api/projects/{id}/deduction-rules/current`。
- `PUT /api/projects/{id}/deduction-rules/current`。
- `GET /api/projects/{id}/deduction-rules/history`。

#### 3. 业务逻辑

- 预扣值用于项目额度冻结与违规惩戒扣减，不能直接改项目总量，只能影响可用额度。
- 当项目剩余额度低于预扣阈值时，系统自动给出预警并限制新处置证关联。

#### 4. 前端实现要点

- 在项目配置区增加额度卡片，展示“总量 / 已用 / 预扣 / 可用”。
- 历史版本列表要能直观看到阈值变化轨迹。

### 3.7 线路配置

#### 1. 数据库设计

- 新增 `biz_project_route` 和 `biz_project_route_point`，保存线路编号、项目、适用场地、启停状态、GeoJSON 轨迹。
- 索引建议：`idx_project_route_status(tenant_id, project_id, status)`。

#### 2. 接口定义

- `GET /api/projects/{id}/routes`：项目线路列表。
- `POST /api/projects/{id}/routes`：新增线路。
- `PUT /api/projects/routes/{routeId}/publish`：发布生效线路。

#### 3. 业务逻辑

- 每个项目允许多条线路，但同一时段只能有一条默认线路，便于偏航判断。
- 线路发布后同步更新缓存，供预警引擎和地图看板直接读取。

#### 4. 前端实现要点

- 在 `ProjectsManagement.tsx` 的“配置”入口中增加线路配置页。
- 前端需要支持地图绘制、导入坐标和回放轨迹三种操作。

### 3.8 违法清单

#### 1. 数据库设计

- 新增 `biz_project_violation`，核心字段为 `project_id/vehicle_id/violation_type/source_event_id/source_model_version/level/disposal_status/occur_time`。
- 索引建议：`idx_project_violation_time(tenant_id, project_id, occur_time)`。

#### 2. 接口定义

- `GET /api/projects/{id}/violations`：分页查询违法清单。
- `GET /api/projects/violations/{id}`：违法详情。
- `POST /api/projects/violations/{id}/close`：关闭或转人工复核。

#### 3. 业务逻辑

- 项目违法清单应聚合预警事件、模型案件和人工录入三类来源，但对外只保留一套状态机。
- 处置状态建议统一为 `PENDING/PROCESSING/CONFIRMED/CLOSED`。

#### 4. 前端实现要点

- 复用项目列表操作中的“违规清单”入口。
- 列表支持按类型、等级、车辆、处置状态过滤，并能跳到违规详情。

### 3.9 报表统计

#### 1. 数据库设计

- 新增 `biz_project_report_snapshot`，按日或月沉淀项目维度的方量、趟次、交款、预警数、违规数。
- 索引建议：`idx_project_report_period(tenant_id, stat_date, stat_type)`。

#### 2. 接口定义

- `GET /api/projects/reports/summary`。
- `GET /api/projects/reports/trend`。
- `GET /api/projects/reports/ranking`。
- `POST /api/projects/reports/export`。

#### 3. 业务逻辑

- 项目报表与日报使用同一快照来源，只是聚合维度不同，避免统计结果冲突。
- 大数据量优先读快照，快照缺失时按需重算并回填。

#### 4. 前端实现要点

- 在项目管理列表页增加“报表统计”抽屉或二级页签。
- 图表配置应保留当前筛选条件，方便导出“所见即所得”。

### 3.10 违规配置

#### 1. 数据库设计

- 新增 `biz_project_violation_rule`，保存 `project_id/rule_code/rule_name/threshold_json/escalation_level/push_template_id/status`。
- 建议按 `project_id + rule_code` 做唯一约束。

#### 2. 接口定义

- `GET /api/projects/{id}/violation-rules`。
- `PUT /api/projects/{id}/violation-rules`。
- `POST /api/projects/{id}/violation-rules/reset`：恢复全局默认。

#### 3. 业务逻辑

- 项目级配置优先级高于系统级配置，但必须保留“继承默认值”能力。
- 规则生效后同步通知预警引擎，避免配置保存成功但运行口径未切换。

#### 4. 前端实现要点

- 项目详情页中与阈值、线路配置放在同一配置中心。
- 前端需要明显区分“项目特化值”和“系统默认值”。

## 4. 跨功能公共约束

### 4.1 数据权限与口径

- 所有项目相关查询都必须显式带 `tenant_id`。
- 交款、处置证、违法清单、报表统计统一以项目主档为事实主键，不能各自再造项目编码。

### 4.2 与其他模块依赖

- `处置证清单` 依赖处置证主表落地。
- `项目日报 / 报表统计` 依赖消纳记录、打卡记录和违规事件统一事实表。
- `违规配置` 需要与预警配置共享规则编码。

### 4.3 实施建议

- 优先把配置类接口做成“详情 + 保存 + 预演”模式，减少反复改表。
- 报表、日报、交款都建议保留异步导出任务接口，避免后续列表页再次重构。

## 5. 验收标准

- 项目列表、项目详情、交款台账、处置证清单、项目日报全部切到真实接口。
- 打卡、位置、预扣值、线路、违规配置均支持查询、编辑、版本回放。
- 项目违法清单和报表统计支持按项目维度筛选、导出和跳转详情。
