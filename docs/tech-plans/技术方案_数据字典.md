# 技术方案 - 数据字典模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`数据字典`
- 本次覆盖的未实现功能共 `1` 个：
  - 新增和编辑

当前仓库现状：

- 前端 `Dictionary.tsx` 仍使用内存 mock 维护字典类型和字典项。
- 后端未发现 `sys_dict_type` / `sys_dict_item` 实体与 controller。
- 字典将成为合同、车辆、预警、消息、小程序等模块的公共依赖，应优先标准化。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `sys_dict_type` | 字典类型主表 |
| `sys_dict_item` | 字典项明细表 |

### 2.2 路由与接口分组

- `/api/dict/types`
- `/api/dict/items`

### 2.3 实施顺序

1. 先建字典类型和字典项主表。
2. 再补列表、详情、新增、编辑与启停。
3. 最后补缓存刷新与前端联动。

---

## 3. 分功能技术方案

### 3.1 新增和编辑

#### 1. 数据库设计

- 新增 `sys_dict_type`，字段建议为 `type_code/type_name/biz_module/status/remark`；新增 `sys_dict_item`，字段建议为 `type_code/item_label/item_value/color_tag/sort_order/status/ext_json`。
- 索引建议：`uk_dict_type_code(type_code)`、`uk_dict_item(type_code, item_value)`。

#### 2. 接口定义

- `GET /api/dict/types` / `POST /api/dict/types` / `PUT /api/dict/types/{id}`。
- `GET /api/dict/items?typeCode=...` / `POST /api/dict/items` / `PUT /api/dict/items/{id}`。

#### 3. 业务逻辑

- 字典类型和字典项都要支持启停与排序，停用后前台不可选但历史数据仍可显示。
- 字典变更后同步刷新缓存，前后端统一按 `typeCode + itemValue` 取值。

#### 4. 前端实现要点

- 把 `Dictionary.tsx` 左侧类型列表和右侧字典项表格全部切到真实接口。
- 新增类型与新增字典项都用抽屉表单，保存后原地刷新当前列表。

## 4. 跨功能公共约束

### 4.1 公共依赖治理

- 字典只承载静态枚举和展示元数据，不承载运行时阈值。
- 模块内若已写死状态枚举，后续统一迁到字典表后再逐页替换。

## 5. 验收标准

- 支持字典类型和字典项的新增、编辑、启停与排序。
- `Dictionary.tsx` 不再依赖本地 mock 数据。
