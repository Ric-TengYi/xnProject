# 技术方案 - 车队管理模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`车队管理`
- 本次覆盖的未实现功能共 `5` 个：
  - 车队信息查询
  - 车队信息维护
  - 车队运输计划
  - 车队调拨申请
  - 油(电)卡管理

当前仓库现状：

- 前端 `FleetManagement.tsx` 只有卡片式 mock 展示，后端完全缺失。
- `VehiclesCards.tsx` 已提供油电卡管理界面骨架，实际仍是静态数据；这部分在需求口径里归属车队管理。
- 车队模块是车辆、司机、运输计划、调拨申请、油电卡的组织层，不能只做静态分组。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_fleet` | 车队主表 |
| `biz_fleet_member` | 车队成员关系，含司机和管理人员 |
| `biz_fleet_vehicle_rel` | 车队与车辆关系 |
| `biz_fleet_transport_plan` | 车队运输计划 |
| `biz_fleet_transfer_apply` | 车队调拨申请 |
| `biz_energy_card` | 油电卡主表 |
| `biz_energy_card_txn` | 油电卡充值/消费流水 |

### 2.2 路由与接口分组

- `/api/fleets`
- `/api/fleet-plans`
- `/api/fleet-transfer-applies`
- `/api/energy-cards`
- `/api/energy-card-transactions`

### 2.3 实施顺序

1. 先做车队主档、成员和车辆关系。
2. 再补运输计划和调拨申请。
3. 最后接入油电卡管理。

---

## 3. 分功能技术方案

### 3.1 车队信息查询

#### 1. 数据库设计

- 新增 `biz_fleet`、`biz_fleet_member`、`biz_fleet_vehicle_rel`，字段建议为 `fleet_code/fleet_name/carrier_org_id/captain_user_id/status`。
- 索引建议：`idx_fleet_org_status(tenant_id, carrier_org_id, status)`。

#### 2. 接口定义

- `GET /api/fleets`：车队列表。
- `GET /api/fleets/{id}`：车队详情。

#### 3. 业务逻辑

- 车队详情应聚合成员数、车辆数、当日出车数和当前状态。
- 查询接口要支持按承运单位、车队状态和负责人筛选。

#### 4. 前端实现要点

- 把 `FleetManagement.tsx` 卡片列表改为真实接口。
- 卡片点击后可进入车队详情页或抽屉。

### 3.2 车队信息维护

#### 1. 数据库设计

- `biz_fleet` 补齐 `contact_mobile/remark/on_duty_status/start_service_date` 等字段。
- 成员和车辆关系表记录生效时间、失效时间和主副岗。

#### 2. 接口定义

- `POST /api/fleets`。
- `PUT /api/fleets/{id}`。
- `POST /api/fleets/{id}/members`。
- `POST /api/fleets/{id}/vehicles`。

#### 3. 业务逻辑

- 维护操作必须校验同一车辆同一时段不能同时归属两个启用车队。
- 解散车队不物理删除，只改状态并保留历史关系。

#### 4. 前端实现要点

- 车队编辑页支持维护基本信息、成员和车辆。
- 前端对即将冲突的车辆或人员给出明确提示。

### 3.3 车队运输计划

#### 1. 数据库设计

- 新增 `biz_fleet_transport_plan`，字段建议为 `fleet_id/plan_date/project_id/site_id/vehicle_target/trip_target/dispatch_status/remark`。
- 按 `fleet_id + plan_date + project_id` 建唯一约束。

#### 2. 接口定义

- `GET /api/fleet-plans`。
- `POST /api/fleet-plans`。
- `PUT /api/fleet-plans/{id}`。

#### 3. 业务逻辑

- 运输计划要能承接项目、场地、车辆目标量，后续可给司机端下发任务。
- 计划状态建议为 `DRAFT/PUBLISHED/EXECUTING/FINISHED/CANCELLED`。

#### 4. 前端实现要点

- 在车队详情中增加“运输计划”页签。
- 列表支持按日期、项目、执行状态查询。

### 3.4 车队调拨申请

#### 1. 数据库设计

- 新增 `biz_fleet_transfer_apply`，字段建议为 `apply_no/source_fleet_id/target_fleet_id/vehicle_id/driver_user_id/reason/approval_status/effective_date`。
- 与审批配置通过 `biz_type=FLEET_TRANSFER` 绑定。

#### 2. 接口定义

- `POST /api/fleet-transfer-applies`。
- `GET /api/fleet-transfer-applies`。
- `POST /api/fleet-transfer-applies/{id}/submit`。

#### 3. 业务逻辑

- 调拨通过后事务内更新车队车辆关系和成员关系。
- 同一车辆存在未完成调拨时禁止重复发起。

#### 4. 前端实现要点

- 车队页增加“调拨申请”入口和审批状态展示。
- 申请表单要实时展示源车队和目标车队现有容量。

### 3.5 油(电)卡管理

#### 1. 数据库设计

- 新增 `biz_energy_card` 与 `biz_energy_card_txn`，字段建议为 `card_no/card_type/carrier_org_id/fleet_id/vehicle_id/balance/status` 与 `txn_type/amount/occur_time/source_type`。
- 按 `card_no` 做唯一约束，流水按 `card_id + occur_time` 建索引。

#### 2. 接口定义

- `GET /api/energy-cards`。
- `POST /api/energy-cards`。
- `POST /api/energy-cards/{id}/recharge`。
- `POST /api/energy-cards/{id}/bind`。

#### 3. 业务逻辑

- 油电卡应同时支持“所属单位维度”和“车队维度”管理，余额不足可触发消息提醒。
- 充值、消费、绑定、解绑全部保留流水，避免只改余额。

#### 4. 前端实现要点

- 复用 `VehiclesCards.tsx` 作为一期油电卡管理页。
- 前端展示余额、累计充值、累计消费、绑定车牌和状态，数据全部改走真实接口。

## 4. 跨功能公共约束

### 4.1 组织层定位

- 车队是运输组织层，成员、车辆、油电卡、运输计划都围绕 `fleet_id` 聚合。
- 车辆和人员的主档仍归车辆/用户域维护，车队只维护关系和计划。

### 4.2 审批与消息联动

- 调拨申请复用审批配置。
- 运输计划发布、油电卡余额预警等信息同步接入消息中心。

## 5. 验收标准

- 车队信息查询、维护、运输计划、调拨申请、油电卡管理全部具备真实接口。
- `FleetManagement.tsx` 与 `VehiclesCards.tsx` 均切到真实数据。
