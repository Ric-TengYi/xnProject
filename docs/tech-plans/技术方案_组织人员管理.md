# 技术方案 - 组织人员管理模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`组织人员管理`
- 本次覆盖的未实现功能共 `2` 个：
  - 人员列表
  - 人员详情

当前仓库现状：

- 后端 `UsersController` 已实现人员新增、编辑、状态变更、角色与组织关系更新。
- `GET /api/users` 当前返回的 `mainOrgName` 为 `null`、`roleNames` 为空列表，`GET /api/users/{id}` 也只返回组织/角色 ID，缺少名称和汇总信息。
- 前端 `Organization.tsx` 已按“组织树 + 人员列表”布局联调真实接口，但页面上仍在消化后端返回的占位字段。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `sys_user` | 人员主表 |
| `sys_org` | 组织主表，用于名称回填与组织树过滤 |
| `sys_role` | 角色主表，用于角色名称聚合 |
| `sys_user_org_rel` | 用户与组织关系 |
| `sys_user_role_rel` | 用户与角色关系 |
| `sys_login_log` | 最近登录时间、登录状态等辅助聚合来源 |

本模块以聚合查询改造为主，不建议新增平行业务主表。

### 2.2 路由与接口分组

- `/api/users`
- `/api/users/{id}`
- `/api/users/{id}/orgs`
- `/api/users/{id}/roles`

### 2.3 实施顺序

1. 先补齐人员列表聚合字段。
2. 再补齐人员详情对象结构和名称回填。
3. 最后优化组织树筛选、角色筛选和最近登录时间展示。

---

## 3. 分功能技术方案

### 3.1 人员列表

#### 1. 数据库设计

- 复用 `sys_user`、`sys_org`、`sys_role`、`sys_user_org_rel`、`sys_user_role_rel`。
- 建议补充索引：
  - `idx_sys_user_tenant_status_org(tenant_id, status, main_org_id)`
  - `idx_sys_user_org_rel_tenant_org_user(tenant_id, org_id, user_id)`
  - `idx_sys_user_role_rel_tenant_user_role(tenant_id, user_id, role_id)`
- 最近登录时间可优先取 `sys_user.last_login_time`，缺失时再回退 `sys_login_log` 聚合。

#### 2. 接口定义

- `GET /api/users`：新增支持 `roleId`、`userType`、`mobile`、`lockStatus` 等筛选。
- 返回字段至少补齐：`mainOrgName`、`roleNames`、`lastLoginTime`、`userType`。
- 如后续需要组织树级联过滤，可补 `orgScope=SELF|CHILDREN` 参数。

#### 3. 业务逻辑

- 列表查询必须在服务端完成组织名称、角色名称聚合，不能继续由前端二次拼接。
- 选中某个组织节点时，建议默认按“组织及下级组织”过滤，避免用户看到空列表误判无数据。
- 列表返回需兼顾性能，角色名称和组织名称建议一次性批量查询再回填，避免 N+1。

#### 4. 前端实现要点

- `Organization.tsx` 的“所属组织”“角色”两列直接消费后端真实字段，不再使用占位值。
- 列表顶部建议补角色筛选、状态筛选和重置按钮。
- 若后端支持组织树级联过滤，左侧树节点切换时应保留其他筛选条件。

### 3.2 人员详情

#### 1. 数据库设计

- 继续复用现有用户、组织、角色关系表，不新增主表。
- 详情聚合建议返回：主组织、所属组织列表、角色列表、最近登录时间、密码重置状态、锁定状态。
- 如后续需要头像、证件、扩展档案，可追加 `sys_user_profile_ext`，但当前阶段非必需。

#### 2. 接口定义

- `GET /api/users/{id}`：返回完整 `UserDetailDto`，其中 `orgs`、`roles` 必须带 `id/code/name`。
- `GET /api/users/{id}/orgs`：返回组织选项列表。
- `GET /api/users/{id}/roles`：返回角色选项列表。

#### 3. 业务逻辑

- 人员详情不应只返回关系 ID，至少要带展示名称，保证编辑弹窗与详情抽屉可直接渲染。
- 主组织必须与 `sys_user.main_org_id` 和 `sys_user_org_rel.is_main=1` 语义保持一致，禁止双主组织。
- 密码、敏感证件号等字段不能明文返回；如需要状态展示，只返回布尔或脱敏值。

#### 4. 前端实现要点

- 建议在 `Organization.tsx` 中补一个人员详情抽屉，同时复用给编辑表单初始化。
- 组织和角色多选控件直接读取详情接口返回的 `orgs`、`roles`，避免再次请求多个接口拼装。
- 详情页应能展示最近登录时间、状态、所属组织、角色标签和基础联系方式。

## 4. 跨功能公共约束

### 4.1 数据权限

- 人员列表和人员详情必须带 `tenant_id` 过滤。
- 平台角色与租户角色在查询层就要做隔离，不能只依赖前端隐藏。

### 4.2 查询性能

- 组织名称、角色名称聚合必须走批量查询或单 SQL 聚合，不要在循环里逐条查表。
- 分页列表优先返回必要字段，详情接口再补足扩展信息。

## 5. 验收标准

- 人员列表可正确展示主组织名称、角色名称、状态和最近登录时间。
- 人员详情可查看完整组织、角色和基础档案信息，不再只返回 ID。
- `Organization.tsx` 的人员列表与详情抽屉都能直接消费真实接口。
