# 技术方案 - 消纳场地模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`消纳场地`
- 本次覆盖的未实现功能共 `7` 个：
  - 场地结算
  - 消纳报表
  - 场地测绘
  - 设备配置
  - 坝体配置
  - 运营配置
  - 人员配置

当前仓库现状：

- `SitesController` 与 `SiteServiceImpl` 已实现列表/详情读取，但 `/api/sites/disposals` 仍明确返回空分页 stub。
- `biz_site` 目前只覆盖基础字段，尚不支持容量、测绘、设备、坝体、运营、安全和值班等配置。
- 前端已有 `SitesManagement.tsx`、`SiteDetail.tsx`、`SitesDisposals.tsx`、`SitesDocuments.tsx` 骨架，除基础信息外其余基本都是 mock。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_site` | 场地主表，补齐租户、场地类型、容量、运营单位、经纬度、启停状态等字段 |
| `biz_site_settlement_profile` | 场地结算维度配置 |
| `biz_site_report_snapshot` | 场地报表快照 |
| `biz_site_survey` | 场地测绘成果与版本 |
| `biz_site_device` | 场地设备台账 |
| `biz_site_dam_config` | 坝体配置与监测规则 |
| `biz_site_operation_config` | 运营配置 |
| `biz_site_staff_rel` | 场地人员配置与班次关系 |

### 2.2 路由与接口分组

- `/api/sites`
- `/api/sites/disposals`
- `/api/sites/settlements`
- `/api/sites/reports/*`
- `/api/sites/surveys`
- `/api/sites/devices`
- `/api/sites/dam-configs`
- `/api/sites/operation-configs`
- `/api/sites/staffings`

### 2.3 实施顺序

1. 升级 `biz_site`，补齐场地主档和详情接口。
2. 打通 `场地测绘 / 设备配置 / 坝体配置 / 运营配置 / 人员配置` 这些基础配置。
3. 随后实现 `消纳报表` 和运营侧 `场地结算`。

---

## 3. 分功能技术方案

### 3.1 场地结算

#### 1. 数据库设计

- 复用合同结算模块的 `biz_settlement_order` / `biz_settlement_item`，在场地域补一张 `biz_site_settlement_profile` 保存结算口径。
- 建议补充 `site_id/pricing_mode/settlement_cycle/rounding_rule` 等场地级结算参数。

#### 2. 接口定义

- `POST /api/sites/{id}/settlements/generate`。
- `GET /api/sites/{id}/settlements`。
- `GET /api/settlements/{settlementId}` 复用结算详情。

#### 3. 业务逻辑

- 场地侧入口主要负责“运营对账”，仍复用统一结算单模型，避免财务和运营两套结算单。
- 生成结算单前要按场地配置校验有效消纳记录、作废记录和重复结算记录。

#### 4. 前端实现要点

- 从 `SiteDetail.tsx` 增加“发起结算”入口，默认带入当前场地。
- 列表页与合同结算模块共用结算详情组件，只切换展示指标。

### 3.2 消纳报表

#### 1. 数据库设计

- 新增 `biz_site_report_snapshot`，沉淀场地日/月维度的容量使用率、车辆数、异常数、结算量。
- 索引建议：`idx_site_report_period(tenant_id, site_id, stat_date, stat_type)`。

#### 2. 接口定义

- `GET /api/sites/reports/summary`。
- `GET /api/sites/reports/trend`。
- `POST /api/sites/reports/export`。

#### 3. 业务逻辑

- 报表优先读取快照；缺失时实时聚合 `biz_disposal_record`、结算单和预警事件回填。
- 场地报表要区分运营量、有效量和异常量三类指标。

#### 4. 前端实现要点

- 在 `SitesManagement.tsx` 和 `SiteDetail.tsx` 增加报表入口。
- 图表支持“日 / 周 / 月”切换，并与场地状态筛选联动。

### 3.3 场地测绘

#### 1. 数据库设计

- 新增 `biz_site_survey`，核心字段建议为 `site_id/survey_version/survey_type/geo_json/capacity_estimate/elevation_data/file_url/status`。
- 同一场地允许多版本测绘成果，但只有一个当前生效版本。

#### 2. 接口定义

- `GET /api/sites/{id}/surveys`。
- `POST /api/sites/{id}/surveys`：上传或录入测绘成果。
- `PUT /api/sites/surveys/{surveyId}/publish`：发布版本。

#### 3. 业务逻辑

- 测绘成果既服务容量计算，也服务围栏、地图和安全检查，不建议拆成多套坐标数据。
- 发布新版本后要同步刷新场地边界缓存。

#### 4. 前端实现要点

- 场地详情页增加地图图层与版本时间轴。
- 前端支持上传测绘文件、手工绘制和查看历史版本差异。

### 3.4 设备配置

#### 1. 数据库设计

- 新增 `biz_site_device`，统一管理摄像头、道闸、地磅、LED、扬尘监测等设备。
- 字段建议包含 `site_id/device_type/device_code/vendor/status/stream_url/ext_json`。

#### 2. 接口定义

- `GET /api/sites/{id}/devices`。
- `POST /api/sites/{id}/devices`。
- `PUT /api/sites/devices/{deviceId}/status`。

#### 3. 业务逻辑

- 设备配置要同时满足“资产台账 + 运行状态 + 外部接入配置”，不要把外部参数散落在系统参数表。
- 场地停用时联动停用设备，但保留历史资产记录。

#### 4. 前端实现要点

- 复用 `SitesManagement.tsx` 卡片中的“监控画面 / 配置设备”入口。
- 设备列表要支持按类型切换，并预留视频流预览区域。

### 3.5 坝体配置

#### 1. 数据库设计

- 新增 `biz_site_dam_config`，字段建议包含 `site_id/dam_section_code/monitor_type/upper_limit/warning_limit/inspection_cycle`。
- 若存在监测点，可再补 `biz_site_dam_sensor` 记录传感器映射。

#### 2. 接口定义

- `GET /api/sites/{id}/dam-configs`。
- `POST /api/sites/{id}/dam-configs`。
- `PUT /api/sites/dam-configs/{configId}`。

#### 3. 业务逻辑

- 坝体配置与安全管理共用同一监测对象编码，便于后续隐患闭环。
- 阈值生效后要可直接触发预警配置中的场地预警规则。

#### 4. 前端实现要点

- 在场地详情页单独提供“坝体配置”页签。
- 支持按坝段展示阈值、巡检周期和最近一次检查结果。

### 3.6 运营配置

#### 1. 数据库设计

- 新增 `biz_site_operation_config`，保存 `site_id/open_time/close_time/queue_limit/accept_source_types/pricing_mode/checkin_required`。
- 建议预留 `holiday_strategy` 和 `night_shift_enabled` 字段。

#### 2. 接口定义

- `GET /api/sites/{id}/operation-configs/current`。
- `PUT /api/sites/{id}/operation-configs/current`。

#### 3. 业务逻辑

- 运营配置决定场地可接收的项目类型、运输来源、夜间作业策略和排队上限。
- 配置变更后直接影响小程序预约、打卡校验和现场设备联动。

#### 4. 前端实现要点

- 场地详情页增加运营设置面板。
- 前端保存后展示“影响范围提示”，提醒哪些业务会联动生效。

### 3.7 人员配置

#### 1. 数据库设计

- 新增 `biz_site_staff_rel`，记录场地负责人、值班员、安全员、巡检员等岗位关系。
- 字段建议为 `site_id/user_id/role_code/shift_code/mobile/on_call_flag/status`。

#### 2. 接口定义

- `GET /api/sites/{id}/staffings`。
- `POST /api/sites/{id}/staffings`。
- `DELETE /api/sites/staffings/{id}`。

#### 3. 业务逻辑

- 人员配置同时服务消息推送、安全检查和审批指派，应优先复用组织用户主数据。
- 一个用户可在多个场地任岗，但同场地同角色同班次不可重复配置。

#### 4. 前端实现要点

- 场地详情页增加值班排班表。
- 支持选择组织用户、设置岗位和快速复制上一班次配置。

## 4. 跨功能公共约束

### 4.1 场地主档口径

- 场地基础信息、测绘、容量、设备、运营配置都必须围绕 `biz_site.id` 作为主键扩展。
- 地图边界、容量、预警阈值不要分别散落在不同模块。

### 4.2 模块联动

- `场地结算` 依赖消纳记录事实表。
- `坝体配置` 与 `安全管理`、`预警配置` 共享阈值与责任人。
- `设备配置` 与 `平台对接` 的视频、地磅接入复用同一设备编码。

## 5. 验收标准

- 场地详情能够展示真实的测绘、设备、坝体、运营和人员配置。
- 场地结算与消纳报表支持按场地维度生成、查询和导出。
- `/api/sites/disposals` 不再返回空分页，而是接入真实消纳记录。
