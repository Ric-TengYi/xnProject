# 技术方案 - 数据看板模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`数据看板`
- 本次覆盖的未实现功能共 `2` 个：
  - 地图展示
  - 消纳场地可视化

当前仓库现状：

- 后端只有 `DashboardController#summary`，返回固定 0 值。
- 前端 `DashboardMap.tsx`、`DashboardSites.tsx`、`DashboardProjects.tsx` 都是可视化骨架和模拟数字。
- 该模块需要同时解决地图数据源、实时快照和刷新策略三件事。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_dashboard_snapshot` | 看板实时快照 |
| `biz_dashboard_map_layer` | 地图图层和标注配置 |
| `biz_site_report_snapshot` | 场地维度可视化指标来源 |
| `biz_project_report_snapshot` | 项目维度可视化指标来源 |

### 2.2 路由与接口分组

- `/api/dashboard/summary`
- `/api/dashboard/map`
- `/api/dashboard/sites`
- `/api/dashboard/projects`

### 2.3 实施顺序

1. 先做看板快照和 summary 接口。
2. 再补地图展示数据。
3. 最后打通场地可视化。

---

## 3. 分功能技术方案

### 3.1 地图展示

#### 1. 数据库设计

- 新增 `biz_dashboard_map_layer`，保存图层编码、图层类型、GeoJSON、样式配置和显示范围。
- `biz_dashboard_snapshot` 中补充在线车辆数、预警数、活跃项目数、场地分布等实时指标。

#### 2. 接口定义

- `GET /api/dashboard/map/markers`。
- `GET /api/dashboard/map/tracks`。
- `GET /api/dashboard/summary`：返回顶部指标。

#### 3. 业务逻辑

- 地图数据按“图层 + 实时点位 + 轨迹回放”三段提供，避免一个接口塞所有数据。
- 对车辆轨迹类数据建议按时间窗查询，不做长期全量拉取。

#### 4. 前端实现要点

- 把 `DashboardMap.tsx` 的左侧分布、右侧地图和底部轨迹条切到真实接口。
- 地图 SDK 选择可后置，但接口先按标准 GeoJSON 输出。

### 3.2 消纳场地可视化

#### 1. 数据库设计

- 复用 `biz_site_report_snapshot`，对看板再补“今日量、排名、容量使用率、预警状态”等字段。
- 如需更高刷新频率，可在 `biz_dashboard_snapshot` 中单独缓存场地看板摘要。

#### 2. 接口定义

- `GET /api/dashboard/sites/cards`。
- `GET /api/dashboard/sites/ranking`。

#### 3. 业务逻辑

- 场地可视化只负责大屏展示，不再承担复杂筛选逻辑；复杂分析跳转统计分析模块。
- 场地卡片排名建议固定按今日量或利用率，避免前端自己重排。

#### 4. 前端实现要点

- 将 `DashboardSites.tsx` 改为真实卡片列表。
- 点击卡片后仍跳转场地详情，实现“看板浏览 -> 业务处理”闭环。

## 4. 跨功能公共约束

### 4.1 刷新策略

- 看板接口要区分秒级快照和分钟级快照，避免所有数据都走实时重算。
- 大屏展示需要明确 `refreshTime`，前端按此展示数据新鲜度。

### 4.2 可视化边界

- 看板只展示摘要指标和可视化数据，不承载复杂编辑能力。
- 地图、项目、场地看板的统计口径必须来自统一快照层。

## 5. 验收标准

- `DashboardController#summary` 返回真实指标。
- 地图展示和场地可视化都能按真实数据刷新。
