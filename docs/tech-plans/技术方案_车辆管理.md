# 技术方案 - 车辆管理模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`车辆管理`
- 本次覆盖的未实现功能共 `6` 个：
  - 车辆保险管理
  - 车辆维修管理
  - 车辆维保计划
  - 车辆维保信息登记
  - 车辆维修申请
  - 车型管理

当前仓库现状：

- 后端只有 `VehiclesController` 空列表/空详情/空新增占位，`biz_vehicle` 实体字段非常薄。
- 前端已有 `VehiclesManagement.tsx`、`VehicleTracking.tsx`、`VehiclesCards.tsx`、`ViolationsList.tsx` 等页面骨架，但保险、维修、维保、车型都没接真实数据。
- 车辆域后续要与处置证、车队、预警、平台对接、小程序联动，建议先补主档与生命周期表。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_vehicle` | 车辆主表，补齐租户、所属单位、车型、状态、定位和证件信息 |
| `biz_vehicle_type` | 车型管理 |
| `biz_vehicle_insurance` | 车辆保险管理 |
| `biz_vehicle_maintenance_plan` | 维保计划 |
| `biz_vehicle_maintenance_record` | 维保信息登记 |
| `biz_vehicle_repair_apply` | 车辆维修申请 |
| `biz_vehicle_repair_order` | 车辆维修管理工单 |

### 2.2 路由与接口分组

- `/api/vehicles`
- `/api/vehicle-types`
- `/api/vehicles/insurances`
- `/api/vehicles/maintenance-plans`
- `/api/vehicles/maintenance-records`
- `/api/vehicles/repair-applies`
- `/api/vehicles/repair-orders`

### 2.3 实施顺序

1. 先升级 `biz_vehicle` 与车型主数据。
2. 再补保险、维保计划和维保登记。
3. 最后补维修申请和维修管理工单。

---

## 3. 分功能技术方案

### 3.1 车辆保险管理

#### 1. 数据库设计

- 新增 `biz_vehicle_insurance`，字段建议为 `vehicle_id/policy_no/insurance_type/insurer_name/coverage_amount/start_date/end_date/status/file_ids`。
- 索引建议：`idx_vehicle_insurance_expire(tenant_id, vehicle_id, end_date)`。

#### 2. 接口定义

- `GET /api/vehicles/insurances`。
- `POST /api/vehicles/{id}/insurances`。
- `PUT /api/vehicles/insurances/{insuranceId}/renew`。

#### 3. 业务逻辑

- 车辆可有交强险、商业险等多险种，按险种分别维护。
- 保险即将到期时直接产生日志或预警事件，供预警配置复用。

#### 4. 前端实现要点

- 车辆详情页增加保险标签和到期提醒。
- 列表支持按险种、到期时间和状态筛选。

### 3.2 车辆维修管理

#### 1. 数据库设计

- 新增 `biz_vehicle_repair_order`，字段建议为 `repair_no/vehicle_id/repair_type/repair_vendor/fault_desc/repair_status/start_time/end_time/cost_amount`。
- 保留与维修申请的 `apply_id` 引用，形成申请到工单闭环。

#### 2. 接口定义

- `GET /api/vehicles/repair-orders`。
- `GET /api/vehicles/repair-orders/{id}`。
- `PUT /api/vehicles/repair-orders/{id}`。

#### 3. 业务逻辑

- 维修管理负责执行中的维修工单，不直接承担申请审批。
- 维修工单完结后可联动车辆状态恢复和维修成本统计。

#### 4. 前端实现要点

- 车辆管理页可新增“维修工单”二级页签。
- 详情中展示故障描述、维修进度、费用和附件。

### 3.3 车辆维保计划

#### 1. 数据库设计

- 新增 `biz_vehicle_maintenance_plan`，字段建议为 `vehicle_id/plan_type/cycle_type/cycle_value/next_execute_date/warning_days/status/template_code`。
- 支持按里程或按日期两种计划类型。

#### 2. 接口定义

- `GET /api/vehicles/maintenance-plans`。
- `POST /api/vehicles/{id}/maintenance-plans`。
- `PUT /api/vehicles/maintenance-plans/{id}`。

#### 3. 业务逻辑

- 维保计划要能自动生成待执行任务，并输出到消息中心。
- 一辆车可配置多类维保计划，但同类型同周期仅保留一条启用计划。

#### 4. 前端实现要点

- 在 `VehiclesManagement.tsx` 中增加“保养到期日”真实来源。
- 计划编辑时提示下次执行时间和预警时间。

### 3.4 车辆维保信息登记

#### 1. 数据库设计

- 新增 `biz_vehicle_maintenance_record`，字段建议为 `vehicle_id/plan_id/execute_time/execute_mileage/service_items/cost_amount/vendor_name/result_status`。
- 计划执行后回写下一次维保时间。

#### 2. 接口定义

- `POST /api/vehicles/maintenance-records`。
- `GET /api/vehicles/maintenance-records`。

#### 3. 业务逻辑

- 维保登记既支持计划内执行，也支持临时维保。
- 登记完成后若存在异常项，可直接转维修申请。

#### 4. 前端实现要点

- 车辆详情页增加维保历史表格。
- 录入表单支持带出计划信息并上传维修单据附件。

### 3.5 车辆维修申请

#### 1. 数据库设计

- 新增 `biz_vehicle_repair_apply`，字段建议为 `apply_no/vehicle_id/applicant_id/fault_desc/urgent_flag/attachment_ids/approval_status/repair_status`。
- 与审批配置通过 `biz_type=VEHICLE_REPAIR` 绑定。

#### 2. 接口定义

- `POST /api/vehicles/{id}/repair-applies`。
- `GET /api/vehicles/repair-applies`。
- `POST /api/vehicles/repair-applies/{id}/submit`。

#### 3. 业务逻辑

- 维修申请发起后进入审批流，通过后才生成维修工单。
- 紧急维修可走特批，但仍保留审批记录。

#### 4. 前端实现要点

- 车辆详情页增加“发起维修申请”按钮。
- 申请列表中同时展示审批状态和后续工单状态。

### 3.6 车型管理

#### 1. 数据库设计

- 新增 `biz_vehicle_type`，字段建议为 `type_code/type_name/load_limit/axle_count/energy_type/status/ext_json`。
- 索引建议：`uk_vehicle_type_code(type_code)`。

#### 2. 接口定义

- `GET /api/vehicle-types`。
- `POST /api/vehicle-types`。
- `PUT /api/vehicle-types/{id}`。

#### 3. 业务逻辑

- 车型是车辆主档和运力分析的公共基础数据，禁止前端写死。
- 停用车型后保留历史车辆引用，但新增车辆不可再选。

#### 4. 前端实现要点

- 车辆新增/编辑表单改为从车型接口加载选项。
- 车型列表支持排序、启停和载重信息展示。

## 4. 跨功能公共约束

### 4.1 生命周期统一

- 保险、维保、维修、违规都围绕同一车辆主档展开，不额外引入临时车辆编码。
- 车辆状态变更应由统一 service 回写，避免各子模块互相覆盖。

### 4.2 与其他模块联动

- 保险到期、维保到期、维修中状态都可触发预警和消息推送。
- 车辆维修申请审批应复用审批配置。
- 车型数据要被车队管理和统计分析共享。

## 5. 验收标准

- 车辆保险、维修、维保计划、维保登记、维修申请、车型管理全部具备真实接口。
- `VehiclesManagement.tsx` 的状态、保养信息等字段改为真实来源。
