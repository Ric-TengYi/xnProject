# Phase 2 第一批模块拆解设计

## 目标

在 Phase 1 权限与 UI 专项验收完成后，按“可交付业务闭环”而不是“零散页面功能点”推进 `xnProject` 的业务整改、回归和验收。第一批任务用于启动稳定执行链，覆盖 UI 收口、权限回归、接口联调、自动化回归和最终验收定义。

## 现状判断

- `phase71-86` 已形成较完整的运行态与 acceptance 证据，说明大量业务模块已经具备真实接口与基本可用性。
- 当前主要问题不再是“模块不存在”，而是“要按多租户 SaaS + 交付验收口径做模块级收口”。
- 用户目标明确为：
  - 先通过用户体系、组织、角色、菜单权限、数据权限、字段限制验收
  - 再对已有业务模块逐个整改、逐个回归、逐个验收
  - 优先解决 UI 边界、交互合理性、权限链路和业务闭环问题

## 三种拆解方式

### 方案 A：按页面逐项拆

- 直接按需求点或页面建卡，例如“项目日报”、“项目报表”、“场地资料”、“处置证关联”。
- 优点：粒度细，适合短平快开发。
- 缺点：容易把一条完整业务链拆散，验收视角碎片化，跨页面问题无人兜底。

### 方案 B：按业务闭环流拆

- 以交付验收视角分组，每张卡覆盖一条完整业务流，内部同时包含 UI、权限、接口、测试与验收。
- 优点：更贴近真实落地；便于定义完成标准；便于后续让 Claude/Codex/Cursor 分工协作。
- 缺点：单张卡范围比页面卡大，需要在描述中写清边界。

### 方案 C：按技术层拆

- 分成“前端整改”“后端整改”“自动化测试”“验收脚本”等技术任务。
- 优点：方便纯技术团队并行。
- 缺点：容易脱离业务闭环，最终仍要重新汇总验收，管理成本最高。

## 推荐方案

采用 **方案 B：按业务闭环流拆**。

原因：

- 现阶段 `xnProject` 已不是从 0 开发，而是要进入“模块收口与交付验收”。
- 你的目标是“多项目自主任务拆解、分配、完成、验收”，Paperclip 上更适合挂载可验收的业务流，而不是纯技术碎任务。
- 后续无论是 Claude 出方案、Codex 做后端、Cursor Gemini 3.1 Pro 做前端、Cursor Auto 做测试，最终都需要回到同一条业务流上汇总。

## 第一批模块分组

### 1. 经营基础与主数据运行链路

覆盖：

- 单位管理
- 项目清单与交款
- 处置证清单与绑定
- 打卡数据
- 消纳信息
- 场地列表/详情/消纳清单
- 车辆与运力基础入口

为什么进第一批：

- 已有 `phase71_business_master_regression` 11/11 通过
- 是后续合同、项目、场地、预警的基础数据底座
- 最适合先做 UI 收口、权限回归和主数据一致性整改

### 2. 合同结算与财务统计链路

覆盖：

- 合同清单/详情
- 合同入账
- 月报统计
- 场地结算
- 合同结算相关消息深链

为什么进第一批：

- 已有 `phase72_contract_settlement_regression` 5/5 通过
- 直接对应交付价值，且对权限、数据口径、导出、消息跳转都敏感
- 是财务和经营验收的主线

### 3. 项目配置与项目统计链路

覆盖：

- 项目日报
- 项目报表
- 打卡配置
- 位置判断
- 出土预扣值
- 线路配置
- 违规围栏
- 项目违规统计

为什么进第一批：

- 已有 `phase73_project_runtime_regression` 7/7 通过
- 用户体系重构后，这类页面最容易暴露数据范围、组织归属和 UI 交互问题
- 后续业务整改会大量围绕项目维度展开

### 4. 场地运营与资料结算链路

覆盖：

- 场地资料
- 场地报表
- 场地设备配置
- 场地运营配置
- 场地人员配置
- 场地创建
- 场地测绘
- 场地安全台账与资料关联

为什么进第一批：

- 已有 `phase39/40/41/42_43/54` 等多个专项 smoke 证据
- 场地端最容易出现 UI 边界、录入校验和角色权限穿透问题
- 该组可直接承接现场交付与运营使用场景

### 5. 预警、事件、安全、消息协同链路

覆盖：

- 预警中心
- 预警配置
- 人工事件
- 安全台账
- 消息中心业务跳转

为什么进第一批：

- 已有 `phase76/77/81` 与 `phase69` 证据
- 这组是“发现问题 -> 派发处理 -> 消息通知 -> 验收闭环”的核心协同链
- 最适合验证多角色、多组织、多租户下的协作真实度

## 每个模块卡片的统一完成定义

每张 Phase 2 业务卡必须同时满足：

1. UI 问题清单补齐并关闭：
   - 列表越界
   - 表单布局不合理
   - 文案、空态、按钮状态、详情抽屉/弹窗交互异常
2. 权限与数据范围回归通过：
   - 平台管理员
   - 租户管理员
   - 普通组织角色
   - 字段级输入限制
   - 菜单、按钮、数据可见范围
3. 业务闭环回归通过：
   - 查询
   - 新增/编辑/提交/审批/驳回/关闭等关键动作
   - 导出、消息跳转、关联详情等补充能力
4. 自动化证据落盘：
   - 复用已有脚本或新增专项回归脚本
   - 测试报告写入 `docs/test-reports`
5. 验收结论明确：
   - 可交付
   - 条件通过
   - 阻塞待修

## 建议执行顺序

1. 经营基础与主数据运行链路
2. 合同结算与财务统计链路
3. 项目配置与项目统计链路
4. 场地运营与资料结算链路
5. 预警、事件、安全、消息协同链路

## 任务编排建议

- Claude：输出每个业务流的整改范围、测试边界、验收标准
- Codex：后端整改、数据口径修正、权限校验、脚本补强
- Cursor Gemini 3.1 Pro：前端页面整改、交互收口、UI 缺陷修复
- Cursor Auto：按模块执行页面回归与关键链路测试
- Claude：基于报告给出最终验收结论

## 结论

Phase 2 第一批不再按零散功能点推进，而是按 5 条业务闭环流推进。Paperclip 看板上应创建 1 张 Phase 2 父任务和 5 张第一批子任务，每张子任务都包含范围、依赖、验收标准和自动化回归要求。
