# 技术方案 - 小程序模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`小程序`
- 本次覆盖的未实现功能共 `6` 个：
  - 出土单位
  - 出土拍照
  - 打卡异常申报
  - 延期申报
  - 问题反馈
  - 司机专属功能

当前仓库现状：

- 仓库当前只覆盖 Web 端，未发现小程序后端接口和移动端代码。
- 需求中的“出土单位”“出土拍照”“异常申报”“司机专属功能”都属于典型移动场景，必须先统一 token、附件和待办模型。
- 小程序能力与项目管理、打卡数据、消息中心、审批配置强耦合，适合按轻量接口统一设计。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `mini_user_binding` | 小程序用户绑定关系 |
| `mini_excavation_org_binding` | 出土单位和用户、项目的可见关系 |
| `mini_excavation_photo` | 出土拍照记录 |
| `mini_checkin_exception_apply` | 打卡异常申报 |
| `mini_delay_apply` | 延期申报 |
| `mini_feedback` | 问题反馈 |
| `mini_driver_task` | 司机任务与专属看板数据 |

### 2.2 路由与接口分组

- `/api/mini/auth/*`
- `/api/mini/excavation-orgs`
- `/api/mini/photos`
- `/api/mini/checkin-exceptions`
- `/api/mini/delay-applies`
- `/api/mini/feedbacks`
- `/api/mini/drivers/*`

### 2.3 实施顺序

1. 先做小程序登录绑定和统一附件能力。
2. 再补出土单位、出土拍照、打卡异常申报和延期申报。
3. 最后补问题反馈和司机专属功能。

---

## 3. 分功能技术方案

### 3.1 出土单位

#### 1. 数据库设计

- 新增 `mini_user_binding` 与 `mini_excavation_org_binding`，保存小程序 openId、用户、组织、项目可见关系。
- 字段建议包含 `user_id/org_id/project_id/role_code/status/last_login_time`。

#### 2. 接口定义

- `POST /api/mini/auth/login`。
- `GET /api/mini/excavation-orgs/current`。
- `GET /api/mini/excavation-orgs/projects`。

#### 3. 业务逻辑

- 小程序身份要与现有用户体系复用同一 `user_id`，不再单独造一套账号。
- 出土单位首页只展示当前用户有权限看到的项目和待办。

#### 4. 前端实现要点

- 移动端首页以“当前单位 + 可用项目 + 待办事项”作为首屏。
- 若未绑定正式账号，返回明确的绑定引导信息。

### 3.2 出土拍照

#### 1. 数据库设计

- 新增 `mini_excavation_photo`，字段建议为 `project_id/site_id/user_id/photo_type/file_id/shoot_time/longitude/latitude/remark/audit_status`。
- 附件文件仍复用统一文件表，拍照记录只保存文件引用和业务元数据。

#### 2. 接口定义

- `POST /api/mini/photos`：上传出土照片。
- `GET /api/mini/photos`：查询我的拍照记录。

#### 3. 业务逻辑

- 拍照上传后要保留拍摄时间和定位，便于后续稽核。
- 若项目配置要求先拍照再出场，可在服务端直接返回校验结果。

#### 4. 前端实现要点

- 移动端支持现场拍照、补充备注和查看最近上传记录。
- 列表默认按最近时间倒序，并可按项目过滤。

### 3.3 打卡异常申报

#### 1. 数据库设计

- 新增 `mini_checkin_exception_apply`，字段建议为 `checkin_record_id/project_id/user_id/exception_type/reason/attachment_ids/status`。
- 建议与 `biz_checkin_abnormal` 建一一或一对多映射，便于复核。

#### 2. 接口定义

- `POST /api/mini/checkin-exceptions`。
- `GET /api/mini/checkin-exceptions`：查询申报历史。

#### 3. 业务逻辑

- 异常申报提交后自动进入审批或人工复核队列，不直接改原始打卡结果。
- 状态建议为 `SUBMITTED/REVIEWING/APPROVED/REJECTED/CLOSED`。

#### 4. 前端实现要点

- 移动端表单应支持选中原打卡记录、填写原因并上传佐证。
- 提交成功后在“我的申报”里查看审核进度。

### 3.4 延期申报

#### 1. 数据库设计

- 新增 `mini_delay_apply`，字段建议为 `biz_type/biz_id/project_id/user_id/requested_end_time/reason/attachment_ids/status`。
- 若与项目管理的延期逻辑联动，保留 `source_channel=MINI`。

#### 2. 接口定义

- `POST /api/mini/delay-applies`。
- `GET /api/mini/delay-applies`。

#### 3. 业务逻辑

- 延期申报只负责发起，审批流仍走统一审批配置。
- 提交后要把关联项目、申请时长和当前状态明确回传给小程序。

#### 4. 前端实现要点

- 移动端提供延期申报表单和历史列表。
- 若材料未齐，直接返回缺失项，不让用户盲提。

### 3.5 问题反馈

#### 1. 数据库设计

- 新增 `mini_feedback`，字段建议为 `feedback_type/project_id/site_id/user_id/content/attachment_ids/status/handler_id/close_time`。
- 支持文本、图片和位置上报。

#### 2. 接口定义

- `POST /api/mini/feedbacks`。
- `GET /api/mini/feedbacks`。
- `POST /api/mini/feedbacks/{id}/close`。

#### 3. 业务逻辑

- 问题反馈走轻工单模式，后续可转事件管理或消息中心。
- 未处理、处理中、已关闭三态先满足一期。

#### 4. 前端实现要点

- 移动端支持反馈提交、进度查询和评价。
- 可选展示最近处理人和处理说明。

### 3.6 司机专属功能

#### 1. 数据库设计

- 新增 `mini_driver_task`，保存司机当日任务、允许路线、关联处置证、待处理预警与待打卡项。
- 表中保留 `driver_user_id/vehicle_id/project_id/site_id/task_date/route_id/permit_id/task_status`。

#### 2. 接口定义

- `GET /api/mini/drivers/home`：司机首页。
- `GET /api/mini/drivers/tasks`。
- `GET /api/mini/drivers/alerts`。

#### 3. 业务逻辑

- 司机端只展示自己车辆和当日任务，不暴露全局数据。
- 预警、路线、打卡和证件有效性要在首页聚合展示，减少司机跳来跳去。

#### 4. 前端实现要点

- 移动端首屏建议展示“任务、路线、证件、预警、打卡”五张卡片。
- 支持快捷进入异常申报和问题反馈。

## 4. 跨功能公共约束

### 4.1 统一身份与文件

- 小程序必须复用现有用户体系与文件存储，不另建账号和附件体系。
- 所有移动端申请单据都保留 `source_channel=MINI`，便于审计。

### 4.2 轻量接口原则

- 移动端接口返回字段尽量聚合，减少多次请求。
- 敏感配置和复杂统计不下发到小程序，只保留任务和待办。

## 5. 验收标准

- 小程序端可完成登录绑定、出土拍照、异常申报、延期申报、问题反馈和司机任务查看。
- 所有移动端单据都能与审批、消息和项目模块联动。
