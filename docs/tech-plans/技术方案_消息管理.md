# 技术方案 - 消息管理模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`消息管理`
- 本次覆盖的未实现功能共 `2` 个：
  - 消息管理
  - 消息推送

当前仓库现状：

- 仓库中未发现消息中心页面和后端发送/投递实现。
- 预警配置、审批流、小程序、平台对接都会触发消息发送需求，不能继续各做各的。
- 建议先建设统一消息中心，再让各业务模块通过事件接入。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `msg_message` | 消息主表，保存消息标题、正文、类型与业务来源 |
| `msg_recipient` | 消息接收人和阅读状态 |
| `msg_channel` | 推送通道配置 |
| `msg_push_task` | 消息推送任务 |
| `msg_push_log` | 通道投递日志 |

### 2.2 路由与接口分组

- `/api/messages`
- `/api/message-channels`
- `/api/message-push-tasks`

### 2.3 实施顺序

1. 先做站内消息主表和接收人模型。
2. 再补通道配置和推送任务。
3. 最后让预警、审批、小程序等模块接入。

---

## 3. 分功能技术方案

### 3.1 消息管理

#### 1. 数据库设计

- 新增 `msg_message` 与 `msg_recipient`，字段建议为 `message_type/biz_type/biz_id/title/content/priority/status/receiver_type/receiver_id/read_time`。
- 消息正文建议同时保留纯文本摘要和富文本 JSON，便于 Web 与小程序共用。

#### 2. 接口定义

- `GET /api/messages`：按类型、已读状态、时间分页查询。
- `GET /api/messages/{id}`：消息详情。
- `PUT /api/messages/{id}/read`：标记已读。

#### 3. 业务逻辑

- 站内消息至少覆盖系统通知、预警通知、审批待办、运营公告四类。
- 一条消息允许多个接收人，阅读状态单独存在 `msg_recipient`，避免为每人复制一条消息正文。

#### 4. 前端实现要点

- 可先新增消息中心页，也可在顶部导航先挂消息抽屉。
- 前端支持按已读/未读、消息类型和业务来源筛选。

### 3.2 消息推送

#### 1. 数据库设计

- 新增 `msg_channel`、`msg_push_task`、`msg_push_log`，字段建议为 `channel_code/channel_type/config_ref/status/retry_count/next_retry_time/result_code/result_message`。
- 消息主表与推送任务之间保持一对多关系，支持同一消息多通道投递。

#### 2. 接口定义

- `GET /api/message-channels` / `PUT /api/message-channels/{id}`。
- `POST /api/message-push-tasks`：创建手工或系统推送任务。
- `GET /api/message-push-tasks/{id}`：查看任务详情和投递日志。

#### 3. 业务逻辑

- 消息推送至少支持站内信、短信、Webhook 三类通道，并预留飞书/钉钉。
- 发送失败必须可重试且保留失败原因，避免业务方只看到“未收到”。

#### 4. 前端实现要点

- 后台页需展示通道状态、任务状态和失败日志。
- 创建推送任务时允许预览最终接收人名单和模板内容。

## 4. 跨功能公共约束

### 4.1 事件驱动接入

- 各业务模块通过统一事件接口入消息中心，不直接拼装推送任务。
- 消息正文模板与接收人解析逻辑统一收口，避免同一业务多套文案。

### 4.2 阅读与投递分离

- 站内阅读状态只影响 `msg_recipient`，不影响外部通道投递日志。
- 推送失败不可直接删除消息主表，只能更新任务状态。

## 5. 验收标准

- 支持站内消息查询、详情、已读和未读统计。
- 支持配置推送通道、创建推送任务并查看投递日志。
