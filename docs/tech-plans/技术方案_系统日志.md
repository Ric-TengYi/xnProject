# 技术方案 - 系统日志模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`系统日志`
- 本次覆盖的未实现功能共 `2` 个：
  - 操作日志
  - 错误日志

当前仓库现状：

- `LoginLogsController` 已实现登录日志查询，登录日志表结构也已在补丁脚本中落地。
- `OperationLogsController` 只有查询骨架，当前 `sys_operation_log` 基线表字段明显不足，仓库中也未发现稳定的操作日志写入链路。
- `SystemController#logs` 只返回空列表，`GlobalExceptionHandler` 也未将错误日志沉淀到数据库。
- 前端 `SystemLogs.tsx` 已有“操作日志 / 登录日志 / 错误日志”三页签，但错误日志仍是纯占位文案。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `sys_login_log` | 已实现登录日志，继续复用 |
| `sys_operation_log` | 操作行为审计日志，需要补齐字段并打通写入 |
| `sys_error_log` | 系统异常与错误日志表 |
| `sys_log_export_task` | 日志导出任务表，承接前端导出按钮 |

### 2.2 路由与接口分组

- `/api/login-logs`
- `/api/operation-logs`
- `/api/error-logs`
- `/api/log-export-tasks`

### 2.3 实施顺序

1. 先升级 `sys_operation_log` 并补写入链路。
2. 再新增 `sys_error_log` 并接入全局异常处理。
3. 最后补日志导出和错误日志前端页签。

---

## 3. 分功能技术方案

### 3.1 操作日志

#### 1. 数据库设计

- 升级 `sys_operation_log`，建议补齐字段：`tenant_id/username/module/operation/method/biz_type/biz_id/request_uri/http_method/result_code/content/ip/user_agent/duration_ms/create_time`。
- 索引建议：
  - `idx_sys_operation_log_tenant_time(tenant_id, create_time)`
  - `idx_sys_operation_log_user_time(user_id, create_time)`
  - `idx_sys_operation_log_module_action(module, action, create_time)`
- 新增 `sys_log_export_task`，保存 `log_type/query_json/status/file_url/creator_id/expire_time`。

#### 2. 接口定义

- `GET /api/operation-logs`：支持按租户、操作人、模块、动作、时间范围分页查询。
- `GET /api/operation-logs/{id}`：查看操作日志详情。
- `POST /api/operation-logs/export`：提交导出任务。
- `GET /api/log-export-tasks/{id}`：查询导出状态。

#### 3. 业务逻辑

- 建议通过 AOP 或拦截器对标注接口统一落操作日志，避免每个 Controller 手写。
- 记录内容应保留“谁在什么时间对哪个资源做了什么动作”，但请求体中的密码、身份证、token 等敏感字段必须脱敏。
- 写日志失败不能阻塞主业务，应走异步写入或失败降级队列。

#### 4. 前端实现要点

- `SystemLogs.tsx` 的操作日志页签应接入模块、时间范围、关键字筛选，而不是只做前端本地过滤。
- 增加日志详情抽屉，展示请求路径、动作、耗时、返回结果码。
- 导出按钮改成异步任务模式，避免大批量日志直接阻塞页面。

### 3.2 错误日志

#### 1. 数据库设计

- 新增 `sys_error_log`，字段建议包含 `tenant_id/trace_id/module/request_uri/http_method/exception_name/error_level/error_fingerprint/message/stack_digest/stack_content/request_snapshot_json/ip/user_agent/handled_status/occur_time`。
- 索引建议：
  - `idx_sys_error_log_tenant_time(tenant_id, occur_time)`
  - `idx_sys_error_log_module_level(module, error_level, occur_time)`
  - `idx_sys_error_log_fingerprint(error_fingerprint, occur_time)`

#### 2. 接口定义

- `GET /api/error-logs`：分页查询错误日志。
- `GET /api/error-logs/{id}`：查询错误详情。
- `PUT /api/error-logs/{id}/handled-status`：标记已处理/忽略。
- `POST /api/error-logs/export`：提交错误日志导出任务。

#### 3. 业务逻辑

- `GlobalExceptionHandler` 在统一返回错误前，先将异常摘要写入 `sys_error_log`。
- 相同堆栈和入口在短时间内可通过 `error_fingerprint` 做归并，减少重复噪音。
- 错误日志必须默认脱敏，尤其不能把明文密码、身份证、token 和上传文件内容直接落库。

#### 4. 前端实现要点

- `SystemLogs.tsx` 的错误日志页签改为真实列表，支持按模块、级别、处理状态、时间范围筛选。
- 列表点击后打开详情抽屉，展示请求上下文、异常摘要和处理备注。
- 错误日志页签要与登录/操作日志保持一致的分页与导出体验。

## 4. 跨功能公共约束

### 4.1 安全与脱敏

- 登录日志、操作日志、错误日志都不得落明文敏感数据。
- 日志表属于高增长表，必须优先按时间建立索引，必要时预留归档与分表策略。

### 4.2 与其他模块依赖

- `系统日志` 的错误日志可作为 `事件管理`、`系统预警` 的人工排查来源，但不能直接替代业务事件。
- 导出任务实现建议复用现有异步导出公共能力，避免再造一套文件生成逻辑。

## 5. 验收标准

- 操作日志具备真实写入、分页查询、详情和导出能力。
- 错误日志可由全局异常处理自动落库，并支持查询与处理状态维护。
- `SystemLogs.tsx` 的三个页签全部接入真实数据。
