# 技术方案 - 平台对接模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`平台对接`
- 本次覆盖的未实现功能共 `3` 个：
  - 政务网数据对接
  - 视频对接
  - 地磅数据对接

当前仓库现状：

- 仓库未发现政务网、视频或地磅的集成模块代码。
- 但前端多个页面已经默认存在视频、称重、外部数据同步等业务预期，后续必须预留标准接口。
- 集成模块必须先设计“连接配置 + 同步任务 + 原始日志 + 对账结果”，否则后面排障会很痛苦。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `int_endpoint` | 外部系统连接配置 |
| `int_sync_task` | 同步任务主表 |
| `int_sync_log` | 同步执行日志 |
| `int_video_device` | 视频设备映射 |
| `int_weighbridge_device` | 地磅设备映射 |
| `int_weighbridge_record` | 地磅原始记录 |

### 2.2 路由与接口分组

- `/api/integrations/gov`
- `/api/integrations/video`
- `/api/integrations/weighbridge`
- `/api/integrations/tasks`

### 2.3 实施顺序

1. 先补外部连接配置和同步任务模型。
2. 再做政务网、视频、地磅三条接入链路。
3. 最后补日志、失败重试和对账能力。

---

## 3. 分功能技术方案

### 3.1 政务网数据对接

#### 1. 数据库设计

- 新增 `int_endpoint` 与 `int_sync_task/int_sync_log`，字段建议为 `integration_code/base_url/auth_type/credential_ref/sync_cursor/status`。
- 政务网数据建议先落中间落地表或 JSON 原文列，避免直接写业务主表。

#### 2. 接口定义

- `POST /api/integrations/gov/sync`。
- `GET /api/integrations/gov/config`。
- `GET /api/integrations/tasks/{taskId}`。

#### 3. 业务逻辑

- 政务网同步默认做增量拉取，游标保存在同步任务中。
- 同步结果分为“拉取成功”“解析失败”“入库失败”三段，便于排障。

#### 4. 前端实现要点

- 可先新增一个平台对接后台页用于配置连接与查看任务。
- 前端展示最近同步时间、同步范围和错误摘要。

### 3.2 视频对接

#### 1. 数据库设计

- 新增 `int_video_device`，保存场地设备、通道号、厂商类型、拉流地址、鉴权参数引用。
- 若需要回放，可扩展 `video_record_index`，但一期可先只做实时流。

#### 2. 接口定义

- `GET /api/integrations/video/devices`。
- `POST /api/integrations/video/devices`。
- `GET /api/integrations/video/devices/{id}/play-token`。

#### 3. 业务逻辑

- 视频对接以“设备映射 + 拉流授权”两层实现，避免前端直接持有敏感鉴权信息。
- 场地设备停用时同步停用视频通道。

#### 4. 前端实现要点

- 场地设备配置页和监控画面入口复用同一视频设备数据。
- 前端拿到播放 token 后再组装播放地址，避免明文泄露。

### 3.3 地磅数据对接

#### 1. 数据库设计

- 新增 `int_weighbridge_device` 与 `int_weighbridge_record`，字段建议包含 `device_code/site_id/record_no/plate_no/gross_weight/tare_weight/net_weight/weigh_time/raw_json/reconcile_status`。
- 按 `device_code + record_no` 做唯一约束，避免重复入库。

#### 2. 接口定义

- `POST /api/integrations/weighbridge/push`：接收地磅推送。
- `GET /api/integrations/weighbridge/records`：查询原始称重记录。
- `POST /api/integrations/weighbridge/reconcile`：人工对账。

#### 3. 业务逻辑

- 地磅记录先落原始表，再由任务异步匹配项目、车辆和场地，避免同步接口阻塞。
- 对账状态建议统一为 `UNMATCHED/MATCHED/ABNORMAL/CONFIRMED`。

#### 4. 前端实现要点

- 场地或车辆详情页可增加地磅记录入口。
- 前端需支持按设备、车牌、时间窗和对账状态过滤。

## 4. 跨功能公共约束

### 4.1 集成审计

- 所有外部调用都必须保留任务记录和失败日志。
- 敏感凭据只存引用，不直接落业务表和前端接口响应。

### 4.2 业务解耦

- 外部同步先落原始数据，再由异步任务入业务主表。
- 视频、地磅、政务网统一以设备或外部对象编码做映射，不直接耦合页面 ID。

## 5. 验收标准

- 政务网、视频、地磅三类接入具备配置、任务、日志和结果查询能力。
- 同步失败可以定位到具体批次和错误原因。
