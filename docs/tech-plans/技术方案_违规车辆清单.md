# 技术方案 - 违规车辆清单模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`违规车辆清单`
- 本次覆盖的未实现功能共 `2` 个：
  - 车辆禁用清单
  - 车辆禁用/解禁

当前仓库现状：

- 后端只有 `biz_vehicle` 基础表与 `VehiclesController` 骨架，尚未形成“违规案件 -> 禁用记录 -> 解禁”的完整链路。
- 前端 `ViolationsList.tsx` 仍使用本地静态数据，页面虽然展示了禁用中、已解禁等状态，但这些状态没有真实后端支撑。
- 系统预警和违规车辆研判模型已经是上游来源，违规车辆清单需要承担“确认违规后的最终处置台账”。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_vehicle` | 车辆主档，补齐禁用状态、最近违规时间等字段 |
| `biz_vehicle_violation_case` | 违规案件表 |
| `biz_vehicle_ban_record` | 禁用/解禁记录表 |

建议对 `biz_vehicle` 增补字段：

- `tenant_id`
- `transport_org_id`
- `ban_status`
- `ban_reason`
- `ban_until`
- `last_violation_time`
- `last_violation_case_id`

### 2.2 路由与接口分组

- `/api/vehicle-violations`
- `/api/vehicle-violations/{id}`
- `/api/vehicle-bans`
- `/api/vehicle-bans/{id}/lift`
- `/api/vehicle-bans/export`

### 2.3 实施顺序

1. 先建设违规案件表，冻结违规确认口径。
2. 再建设禁用清单和禁用/解禁动作。
3. 最后与车辆管理、系统预警、车队管理完成联动。

---

## 3. 分功能技术方案

### 3.1 车辆禁用清单

#### 1. 数据库设计

- 新增 `biz_vehicle_violation_case`，核心字段建议为 `tenant_id/case_no/vehicle_id/plate_no/violation_type/source_alert_id/source_model_version/project_id/site_id/occur_time/confirm_status/penalty_type/penalty_result`。
- 新增 `biz_vehicle_ban_record`，核心字段建议为 `tenant_id/vehicle_id/ban_no/case_id/ban_type/ban_start_time/ban_end_time/status/reason/operator_id/operator_name`。
- 索引建议：
  - `uk_vehicle_violation_case_no(tenant_id, case_no)`
  - `idx_vehicle_violation_vehicle_time(tenant_id, vehicle_id, occur_time)`
  - `idx_vehicle_ban_status_time(tenant_id, status, ban_start_time)`

#### 2. 接口定义

- `GET /api/vehicle-bans`：按车牌号、违规类型、状态、时间范围分页查询禁用清单。
- `GET /api/vehicle-violations/{id}`：查看违规案件详情。
- `POST /api/vehicle-bans/export`：导出禁用清单。

#### 3. 业务逻辑

- “禁用清单”应以已确认违规案件为来源，不建议直接从预警流水表生禁用状态。
- 清单需同时支持“当前禁用中”和“历史已解禁”两种视图，避免只保留当前状态丢失历史记录。
- 车辆维度需能看到违规类型、违规地点、处罚结果、禁用期限和关联运输单位。

#### 4. 前端实现要点

- `ViolationsList.tsx` 的列表、筛选、导出全部切到真实接口。
- 列表支持车牌号、违规类型、处理状态、时间范围筛选，并能切换“全部 / 当前禁用 / 已解禁”。
- 点击“详情”后应展示违规案件来源、预警记录、处罚结果和禁用历史。

### 3.2 车辆禁用/解禁

#### 1. 数据库设计

- 复用 `biz_vehicle_ban_record` 记录每一次禁用、延长禁用和提前解禁动作。
- `biz_vehicle` 主表同步维护当前禁用状态，保证车辆列表、项目校验、场地入场校验能快速读取。
- 索引建议：`idx_vehicle_ban_vehicle_status(tenant_id, vehicle_id, status)`。

#### 2. 接口定义

- `POST /api/vehicle-bans`：发起车辆禁用。
- `POST /api/vehicle-bans/{id}/lift`：执行车辆解禁。
- `GET /api/vehicle-bans/{id}`：查看禁用记录详情。
- `POST /api/vehicle-bans/{id}/extend`：延长禁用期限。

#### 3. 业务逻辑

- 只有已确认违规案件才允许发起禁用，且必须记录禁用依据和期限。
- 禁用时事务内同步更新 `biz_vehicle.ban_status`，并拦截后续项目绑定、进场校验或处置证关联。
- 解禁支持两种方式：到期自动解禁、人工提前解禁；两种方式都必须留痕。

#### 4. 前端实现要点

- `ViolationsList.tsx` 中“去处理”“提前解禁”等按钮改为真实动作，并弹出确认与填写原因表单。
- 禁用中车辆需要高亮展示剩余禁用时长。
- 解禁后列表应自动刷新，同时允许跳转到车辆详情页查看恢复状态。

## 4. 跨功能公共约束

### 4.1 与上游联动

- `系统预警` 和 `违规车辆研判模型` 负责发现风险，`违规车辆清单` 负责沉淀确认后的最终处置台账。
- 若某条预警最终被判定为误报，不应进入禁用清单，只保留在预警处置记录中。

### 4.2 与车辆主档一致性

- 车辆当前禁用状态必须以 `biz_vehicle` 主表为准，历史过程以 `biz_vehicle_ban_record` 为准。
- 禁用或解禁动作都要同步影响车辆管理、车队管理和场地校验链路。

## 5. 验收标准

- 违规车辆清单支持真实列表、详情、筛选和导出。
- 车辆禁用、延长禁用、提前解禁都可真实落库并留痕。
- 车辆主档状态与禁用记录保持一致，前端 `ViolationsList.tsx` 完成真实化。
