# 技术方案 - 审核审批配置模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`审核审批配置`
- 本次覆盖的未实现功能共 `1` 个：
  - 办事材料配置

当前仓库现状：

- `ApprovalActorRulesController` 及对应 service 已可完成审批参与人规则 CRUD，是当前可复用的审批底座。
- `ApprovalConfig.tsx` 中“材料配置”目前只是按钮占位，办事材料模板、提交校验、材料回显都未实现。
- 建议把材料配置做成审批体系的补充层，而不是另起一套审批系统。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `sys_approval_actor_rule` | 现有审批参与人规则表，继续作为流程主体配置 |
| `sys_approval_material_template` | 审批事项的办事材料模板 |
| `sys_approval_material_item` | 模板下的材料项、必填规则与校验规则 |
| `biz_approval_material_submission` | 业务单据提交的材料实例 |

### 2.2 路由与接口分组

- `/api/approval-actor-rules`
- `/api/approval-material-templates`
- `/api/approval-material-submissions`

### 2.3 实施顺序

1. 先新增材料模板与材料项模型。
2. 再把模板绑定到现有 `processKey` 与节点编码。
3. 最后补提交时的材料校验和前端配置页。

---

## 3. 分功能技术方案

### 3.1 办事材料配置

#### 1. 数据库设计

- 新增 `sys_approval_material_template` 与 `sys_approval_material_item`，字段建议为 `process_key/node_code/material_code/material_name/material_type/required_flag/rule_json/sort_order/status`。
- 业务侧通过 `biz_approval_material_submission` 保存每次提交流程的材料快照与校验状态。

#### 2. 接口定义

- `GET /api/approval-material-templates`：按审批事项查询模板。
- `POST /api/approval-material-templates`：新增或保存模板。
- `POST /api/approval-material-submissions/validate`：校验某业务单据材料是否齐全。

#### 3. 业务逻辑

- 模板配置支持“发起节点”和“审批节点”两类材料要求，并允许按条件动态显隐。
- 业务单据提交前先做材料校验，未通过时直接返回缺失项，避免进入流程后再打回。

#### 4. 前端实现要点

- 复用 `ApprovalConfig.tsx` 当前“材料配置”入口，新增模板编辑抽屉。
- 前端要支持材料项排序、是否必传、附件示例和校验提示词维护。

## 4. 跨功能公共约束

### 4.1 复用现有审批底座

- 材料模板必须围绕现有 `processKey` 运转，不再自定义另一套审批事项编码。
- 提交校验结果与审批流状态解耦，只负责准入，不负责审批结果。

## 5. 验收标准

- 审批事项可维护办事材料模板和材料项。
- 业务单据提交前能返回明确的材料校验结果。
- `ApprovalConfig.tsx` 的“材料配置”按钮接入真实能力。
