# 技术方案 - 预警配置模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`预警配置`
- 本次覆盖的未实现功能共 `3` 个：
  - 阈值配置
  - 预警围栏
  - 推送配置

当前仓库现状：

- 后端只有 `AlertsController#list` 返回空列表，未发现阈值、围栏和推送配置实体。
- 前端 `AlertConfig.tsx` 已有阈值、围栏、推送三段表单骨架，但没有真实数据源。
- 该模块会被违规车辆研判模型、消息管理、安全管理和数据看板同时依赖，必须先做统一规则模型。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_alert_rule` | 预警阈值规则主表 |
| `biz_alert_rule_target` | 规则作用范围，支持项目、场地、车队、车辆等目标 |
| `biz_alert_fence` | 预警围栏与空间规则 |
| `biz_alert_push_rule` | 预警推送规则与接收人配置 |

### 2.2 路由与接口分组

- `/api/alert-rules`
- `/api/alert-fences`
- `/api/alert-push-rules`

### 2.3 实施顺序

1. 先落统一预警规则主表，把阈值、等级、作用域模型固定下来。
2. 再做围栏配置，把地理范围与规则绑定。
3. 最后补推送规则，把预警真正发出去。

---

## 3. 分功能技术方案

### 3.1 阈值配置

#### 1. 数据库设计

- 新增 `biz_alert_rule`，字段建议为 `rule_code/rule_name/rule_scene/metric_code/threshold_json/level/status/tenant_id`。
- 若同一规则需要按项目或场地覆盖，使用 `biz_alert_rule_target` 记录作用目标。

#### 2. 接口定义

- `GET /api/alert-rules`。
- `POST /api/alert-rules`。
- `PUT /api/alert-rules/{id}`。
- `PUT /api/alert-rules/{id}/status`。

#### 3. 业务逻辑

- 阈值规则要支持全局默认值和局部覆盖值，并保留优先级。
- 规则发布后输出标准配置快照，供预警引擎读取，避免直接查业务表。

#### 4. 前端实现要点

- 将 `AlertConfig.tsx` 的场地预警和车辆预警表单切到真实接口。
- 前端要区分“系统默认”和“自定义覆盖”。

### 3.2 预警围栏

#### 1. 数据库设计

- 新增 `biz_alert_fence`，保存 `fence_code/fence_name/fence_type/geo_json/buffer_meters/biz_scope/status`。
- 与预警规则通过 `rule_code` 或关联表绑定，便于一条规则套多个围栏。

#### 2. 接口定义

- `GET /api/alert-fences`。
- `POST /api/alert-fences`。
- `POST /api/alert-fences/{id}/preview`：坐标预演。

#### 3. 业务逻辑

- 围栏既能用于禁行、停留超时，也能用于入场、离场校验；不要按场景拆成多套表。
- 围栏发布后需要同步刷新地图缓存与模型配置快照。

#### 4. 前端实现要点

- 在 `AlertConfig.tsx` 增加地图绘制与命中预演。
- 支持按项目、场地、道路类型快速筛选围栏。

### 3.3 推送配置

#### 1. 数据库设计

- 新增 `biz_alert_push_rule`，字段建议为 `rule_code/channel_type/receiver_type/receiver_expr/template_code/escalation_minutes/status`。
- 推送配置可单独维护，也可挂在阈值规则下做子配置。

#### 2. 接口定义

- `GET /api/alert-push-rules`。
- `POST /api/alert-push-rules`。
- `PUT /api/alert-push-rules/{id}`。

#### 3. 业务逻辑

- 推送规则至少支持站内信、短信、Webhook 三类通道，并预留后续飞书/钉钉。
- 高等级预警允许按时间升级和重复提醒，但要做去重防抖。

#### 4. 前端实现要点

- 复用 `AlertConfig.tsx` 现有推送规则 UI。
- 前端支持按预警等级预览推送对象和模板内容。

## 4. 跨功能公共约束

### 4.1 统一规则编码

- 阈值、围栏、推送三张表都要围绕统一 `rule_code` 展开。
- 事件表只记录命中的 `rule_code` 和配置快照版本，不反向依赖配置表结构。

### 4.2 联动要求

- 预警规则发布后要同步通知违规研判模型和消息中心。
- 地图和围栏数据必须能被数据看板直接消费。

## 5. 验收标准

- 阈值、围栏、推送三类配置支持列表、编辑、启停与预演。
- 配置生效后预警事件能回溯命中的规则与版本。
- `AlertConfig.tsx` 不再依赖默认值和静态枚举。
