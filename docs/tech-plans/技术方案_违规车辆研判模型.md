# 技术方案 - 违规车辆研判模型模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`违规车辆研判模型`
- 本次覆盖的未实现功能共 `2` 个：
  - 模型配置
  - 研判车辆处置

当前仓库现状：

- `AlertsMonitor.tsx` 文案上已出现“研判模型运行中”和“立即研判处置”，但后端没有模型配置与处置工单实现。
- 现有 `biz_alert_event` 只能保存简单标题、等级和关联对象，无法表达模型版本、证据链和研判结论。
- 建议把模型配置与案件处置一次性设计成闭环，后续预警事件只做触发入口。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_violation_model` | 违规研判模型主表 |
| `biz_violation_model_version` | 模型版本与参数快照 |
| `biz_violation_case` | 模型输出的违规案件 |
| `biz_violation_case_action` | 案件处置动作与流转日志 |

### 2.2 路由与接口分组

- `/api/violation-models`
- `/api/violation-cases`

### 2.3 实施顺序

1. 先固定模型配置与版本表结构。
2. 再实现案件生成与处置流。
3. 最后把预警中心页面切到模型案件数据。

---

## 3. 分功能技术方案

### 3.1 模型配置

#### 1. 数据库设计

- 新增 `biz_violation_model` 与 `biz_violation_model_version`，字段建议为 `model_code/model_name/model_type/scene/feature_json/threshold_json/status/version_no`。
- 每次发布配置都生成新版本，预警事件和案件只引用版本号。

#### 2. 接口定义

- `GET /api/violation-models`。
- `POST /api/violation-models`。
- `PUT /api/violation-models/{id}/publish`：发布新版本。

#### 3. 业务逻辑

- 模型配置既可以是规则引擎，也可以是阈值模型，字段设计应支持表达式和参数 JSON。
- 发布后保留旧版本，保证历史案件可以按原模型回放。

#### 4. 前端实现要点

- 预警中心右上角增加模型配置入口。
- 前端展示当前运行版本、最近发布时间和影响场景。

### 3.2 研判车辆处置

#### 1. 数据库设计

- 新增 `biz_violation_case`，核心字段建议为 `case_no/vehicle_id/project_id/site_id/source_event_id/model_version_id/risk_level/judge_result/disposal_status/evidence_json`。
- 新增 `biz_violation_case_action` 记录派单、确认、驳回、拉黑、解除等动作。

#### 2. 接口定义

- `GET /api/violation-cases`：分页查询案件。
- `GET /api/violation-cases/{id}`：案件详情。
- `POST /api/violation-cases/{id}/actions`：提交处置动作。

#### 3. 业务逻辑

- 案件状态建议统一为 `PENDING/ASSIGNED/CONFIRMED/REJECTED/CLOSED`。
- 处置动作要能回写车辆状态、消息推送和违规清单，但案件本身保持审计闭环。

#### 4. 前端实现要点

- 把 `AlertsMonitor.tsx` 中“立即研判处置”按钮切到案件处置抽屉。
- 案件详情需展示命中规则、证据时间线和处置记录。

## 4. 跨功能公共约束

### 4.1 版本可回放

- 模型参数变化不能直接覆盖历史案件，必须通过版本表隔离。
- 案件详情必须能追溯到命中的模型版本、规则和证据。

### 4.2 模块联动

- 案件处置结果同步给项目违法清单、车辆管理和消息管理。
- 模型输入数据默认来自预警事件、打卡数据、线路配置和围栏命中结果。

## 5. 验收标准

- 模型配置支持新增、发布、停用和版本回看。
- 违规案件支持查询、详情查看和处置动作闭环。
- `AlertsMonitor.tsx` 的核心列表能够切到真实模型案件。
