# 技术方案 - 单位管理模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`单位管理`
- 本次覆盖的未实现功能共 `3` 个：
  - 建设单位
  - 施工单位
  - 运输单位

当前仓库现状：

- 后端已存在 `/api/tenants`、`/api/tenants/{id}`、`/api/tenants/{id}/summary` 等通用租户接口，可按 `tenantType` 做基础筛选。
- `xngl-service/docs/user-system/00-shared-model.md` 已冻结 `tenant_type`，但当前模型只能表达“租户边界”，还不能完整表达建设单位、施工单位、运输单位的业务属性与资质状态。
- 前端未发现独立的单位管理页面，说明当前“单位管理”仍停留在用户体系基础租户管理阶段，尚未形成业务化主数据台账。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `sys_tenant` | 继续作为单位主表，补齐业务单位分类、信用代码、法人、资质状态等字段 |
| `sys_tenant_qualification` | 单位资质与证照台账 |
| `sys_tenant_contact` | 单位联系人与对接人信息 |

建议对 `sys_tenant` 增补字段：

- `biz_unit_type`：`CONSTRUCTION_OWNER / CONSTRUCTION_CONTRACTOR / TRANSPORT_COMPANY`
- `credit_code`
- `legal_person`
- `license_status`
- `qualification_expire_time`
- `service_scope`
- `cooperation_status`

### 2.2 路由与接口分组

- `/api/units`
- `/api/units/construction-owners`
- `/api/units/construction-contractors`
- `/api/units/transport-companies`
- `/api/units/{id}/qualifications`
- `/api/units/{id}/summary`

### 2.3 实施顺序

1. 先升级 `sys_tenant`，冻结“租户边界”和“业务单位分类”的分工。
2. 再补单位资质、联系人和汇总信息。
3. 最后分别落地建设单位、施工单位、运输单位三个业务视图与筛选项。

## 3. 分功能技术方案

### 3.1 建设单位

#### 1. 数据库设计

- 复用 `sys_tenant`，要求 `biz_unit_type=CONSTRUCTION_OWNER`。
- 新增 `sys_tenant_qualification`，核心字段建议为 `tenant_id/cert_type/cert_no/issue_org/effective_date/expire_date/status/file_url`。
- 索引建议：
  - `idx_sys_tenant_unit_type_status(biz_unit_type, status)`
  - `uk_sys_tenant_credit_code(credit_code)`
  - `idx_sys_tenant_qualification_tenant_status(tenant_id, status, expire_date)`

#### 2. 接口定义

- `GET /api/units/construction-owners`：分页查询建设单位。
- `POST /api/units/construction-owners`：新增建设单位。
- `GET /api/units/{id}`：查看单位详情。
- `GET /api/units/{id}/qualifications`：查看资质证照。
- `PUT /api/units/{id}/status`：启停用单位。

#### 3. 业务逻辑

- 建设单位是项目和合同的主关联方，必须支持按单位名称、统一社会信用代码、状态快速检索。
- 同一信用代码在逻辑上只能对应一个有效单位主档，避免项目、合同引用分裂。
- 建设单位详情页需要能直接看到关联项目数、关联合同数和证照到期提示。

#### 4. 前端实现要点

- 建议新增 `UnitsManagement.tsx`，建设单位作为一个固定页签。
- 列表页支持“证照即将到期”“已停用”“有关联项目”三个快捷筛选。
- 单位详情页应展示基础信息、资质证照、关联项目和合同摘要。

### 3.2 施工单位

#### 1. 数据库设计

- 复用 `sys_tenant`，要求 `biz_unit_type=CONSTRUCTION_CONTRACTOR`。
- 继续复用 `sys_tenant_qualification`、`sys_tenant_contact`。
- 如需沉淀项目合作关系，可新增 `biz_project_unit_rel` 保存 `project_id/tenant_id/rel_type/start_time/end_time/status`。

#### 2. 接口定义

- `GET /api/units/construction-contractors`：分页查询施工单位。
- `POST /api/units/construction-contractors`：新增施工单位。
- `PUT /api/units/{id}`：编辑施工单位。
- `GET /api/units/{id}/summary`：查看项目合作汇总。

#### 3. 业务逻辑

- 施工单位与建设单位共用一个主数据模型，但必须在业务口径上可区分，否则项目、合同、处置证关联时会混淆角色。
- 施工单位详情建议展示在建项目、历史项目、车辆接入数量，便于后续对接车辆和车队模块。
- 若一个单位同时承担建设和施工两种角色，可通过 `biz_unit_type` 多值化或 `biz_unit_role_rel` 扩展，不建议复制主档。

#### 4. 前端实现要点

- 页面结构与建设单位共用列表组件，只切换默认筛选和表头文案。
- 详情页增加“合作项目”区块，可从项目详情页带参跳转。
- 单位新增表单中需要明确标识“单位类型”和“业务角色”。

### 3.3 运输单位

#### 1. 数据库设计

- 复用 `sys_tenant`，要求 `biz_unit_type=TRANSPORT_COMPANY`。
- 在 `sys_tenant_qualification` 中增加运输资质类型，如 `ROAD_TRANSPORT_LICENSE`、`SAFETY_PRODUCTION_LICENSE`。
- 若后续需要快速汇总车辆与车队数量，可补 `biz_transport_unit_snapshot` 日快照表。

#### 2. 接口定义

- `GET /api/units/transport-companies`：分页查询运输单位。
- `POST /api/units/transport-companies`：新增运输单位。
- `GET /api/units/{id}/summary`：返回车辆数、车队数、违规数、禁用车辆数。
- `GET /api/units/{id}/qualifications`：查看运输资质。

#### 3. 业务逻辑

- 运输单位是车辆、车队、违规车辆清单的上游主体，必须保留单位与车辆/车队的稳定主键关系。
- 运输单位状态变更时，需要同步提示是否存在运行中车辆或有效处置证，避免直接停用造成业务断链。
- 运输资质到期应可直接触发系统预警，为后续“系统预警”和“违规车辆清单”提供来源数据。

#### 4. 前端实现要点

- 运输单位列表增加车辆数、车队数、违规车辆数等经营指标列。
- 详情页支持跳转到 `VehiclesManagement.tsx` 与 `ViolationsList.tsx` 的带参视图。
- 资质到期、单位停用应在列表和详情页同时高亮提示。

## 4. 跨功能公共约束

### 4.1 主数据唯一口径

- “租户”继续承担账号和数据隔离边界，“单位管理”负责业务主数据表达，两者必须共用同一主键。
- 建设单位、施工单位、运输单位不应各自再建一套平行主表。

### 4.2 与其他模块依赖

- `项目管理`、`合同结算`、`车辆管理`、`车队管理` 都应通过单位主档做关联，不直接存自由文本单位名。
- `系统预警` 可直接复用单位资质到期、停用状态等事实数据。

## 5. 验收标准

- 建设单位、施工单位、运输单位三个业务视图可独立查询、详情查看和状态维护。
- 单位详情能看到资质、联系人以及关联项目/车辆等摘要信息。
- 单位主数据与租户主档保持一套主键，不出现重复主档。
