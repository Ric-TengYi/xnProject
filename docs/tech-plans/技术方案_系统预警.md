# 技术方案 - 系统预警模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`系统预警`
- 本次覆盖的未实现功能共 `5` 个：
  - 场地预警
  - 项目预警
  - 人员预警
  - 车辆预警
  - 合同预警

当前仓库现状：

- 后端只有 `AlertsController#list` 空实现，`biz_alert_event` 也仅保留 `title/level/relatedId/relatedType/status` 等最小字段。
- 预警配置模块已单独输出技术方案，说明“规则配置”与“预警实例”应拆分建设。
- 前端 `AlertsMonitor.tsx` 仍使用本地定时生成的 mock 告警数据，场地和项目页签也未接真实查询。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_alert_event` | 预警实例事实表，统一承接各类预警 |
| `biz_alert_handle_log` | 预警处置记录 |
| `biz_alert_target_snapshot` | 预警发生时的业务对象快照 |

建议对 `biz_alert_event` 补齐字段：

- `tenant_id`
- `alert_no`
- `alert_type`
- `rule_code`
- `rule_version`
- `target_type`
- `target_id`
- `project_id`
- `site_id`
- `vehicle_id`
- `user_id`
- `contract_id`
- `alert_level`
- `alert_status`
- `occur_time`
- `resolve_time`
- `latest_position_json`
- `snapshot_json`

### 2.2 路由与接口分组

- `/api/alerts`
- `/api/alerts/sites`
- `/api/alerts/projects`
- `/api/alerts/users`
- `/api/alerts/vehicles`
- `/api/alerts/contracts`
- `/api/alerts/{id}`
- `/api/alerts/{id}/handle`
- `/api/alerts/{id}/close`

### 2.3 实施顺序

1. 先升级 `biz_alert_event`，打通统一列表和详情。
2. 再按目标对象维度拆出 5 类预警查询入口。
3. 最后补处置、关闭、统计和前端看板。

## 3. 分功能技术方案

### 3.1 场地预警

#### 1. 数据库设计

- 复用 `biz_alert_event`，要求 `target_type=SITE`。
- 场地预警常见规则包括超时停留、禁入禁出、场地容量异常、排队拥堵等。
- 索引建议：`idx_alert_site_time(tenant_id, target_type, site_id, occur_time)`。

#### 2. 接口定义

- `GET /api/alerts/sites`：分页查询场地预警。
- `GET /api/alerts/sites/summary`：返回场地维度待处置数量与等级分布。

#### 3. 业务逻辑

- 场地预警以场地为主对象，但允许同时携带车辆、项目等次级关联对象。
- 同一辆车在同一场地短时间重复触发同类规则时，建议做事件合并，避免列表刷屏。

#### 4. 前端实现要点

- `AlertsMonitor.tsx` 的“场地预警”页签改走真实接口。
- 列表项建议展示场地名称、预警类型、等级、最近车辆、发生时间和处置状态。

### 3.2 项目预警

#### 1. 数据库设计

- 复用 `biz_alert_event`，要求 `target_type=PROJECT`。
- 项目预警可承接日报异常、打卡异常、路线异常、额度预扣不足等规则结果。
- 索引建议：`idx_alert_project_time(tenant_id, target_type, project_id, occur_time)`。

#### 2. 接口定义

- `GET /api/alerts/projects`：分页查询项目预警。
- `GET /api/alerts/projects/summary`：返回项目维度的风险排名与数量统计。

#### 3. 业务逻辑

- 项目预警应直接关联项目主档，便于从项目详情页反查未处理风险。
- 同一个项目下的不同规则命中要能并列展示，不能只保留一条最新事件覆盖历史。

#### 4. 前端实现要点

- 项目预警列表支持按项目、规则类型、状态和时间范围筛选。
- 支持从 `ProjectsManagement.tsx` 或 `ProjectDetail.tsx` 带参跳转到项目预警视图。

### 3.3 人员预警

#### 1. 数据库设计

- 复用 `biz_alert_event`，要求 `target_type=USER`。
- 预警来源可包括未按规定打卡、人员实名异常、黑名单命中、连续违规等。
- 索引建议：`idx_alert_user_time(tenant_id, target_type, user_id, occur_time)`。

#### 2. 接口定义

- `GET /api/alerts/users`：分页查询人员预警。
- `GET /api/alerts/users/{userId}/trend`：查看人员近一段时间预警趋势。

#### 3. 业务逻辑

- 人员预警必须关联统一的用户主档，避免只存姓名字符串导致无法回溯。
- 若人员预警来源于打卡数据或小程序申报，需要保留 `source_record_id` 与规则版本号，便于复核。

#### 4. 前端实现要点

- 人员预警页可作为 `AlertsMonitor.tsx` 的新增页签或筛选维度。
- 列表中展示人员姓名、所属组织、预警类型、项目/场地上下文与处理状态。

### 3.4 车辆预警

#### 1. 数据库设计

- 复用 `biz_alert_event`，要求 `target_type=VEHICLE`。
- 车辆预警是当前预警中心的一期重点，建议额外建立索引 `idx_alert_vehicle_status_time(tenant_id, target_type, vehicle_id, alert_status, occur_time)`。
- 可选新增 `biz_alert_target_snapshot` 存储车牌号、所属项目、定位等快照，避免主档变更后历史失真。

#### 2. 接口定义

- `GET /api/alerts/vehicles`：分页查询车辆预警。
- `GET /api/alerts/vehicles/top-risk`：返回高风险车辆/车队排名。
- `POST /api/alerts/{id}/handle`：登记研判或人工处置结果。

#### 3. 业务逻辑

- 车辆预警应承接偏航、超速、禁区通行、未打卡入场、证件异常等规则命中结果。
- 一条车辆预警在处置完成前需保持独立实例，不能被新告警覆盖，以便后续进入“违规车辆清单”。
- 若确认构成违规，可直接关联到 `违规车辆清单` 模块的案件或禁用记录。

#### 4. 前端实现要点

- `AlertsMonitor.tsx` 当前主视图就是车辆预警，一期优先把该页签真实化。
- 高风险车队榜单和动态围栏状态卡片可先消费聚合接口，不建议由前端硬编码模拟数据。

### 3.5 合同预警

#### 1. 数据库设计

- 复用 `biz_alert_event`，要求 `target_type=CONTRACT`。
- 合同预警可覆盖合同到期、结算超期、额度不足、资质过期导致合同受阻等场景。
- 索引建议：`idx_alert_contract_time(tenant_id, target_type, contract_id, occur_time)`。

#### 2. 接口定义

- `GET /api/alerts/contracts`：分页查询合同预警。
- `GET /api/alerts/contracts/summary`：返回待处理数、到期数、超期数等摘要。

#### 3. 业务逻辑

- 合同预警应直接关联合同主档，并能反查项目、场地、单位等上下文。
- 合同到期类预警建议支持“提前 N 天触发”，规则参数来自预警配置模块。

#### 4. 前端实现要点

- 合同预警可以先作为列表筛选维度落地，不必一期单独开页面。
- 详情中需要展示关联合同编号、项目名称、到期时间和当前处理人。

## 4. 跨功能公共约束

### 4.1 规则与实例分离

- `预警配置` 负责规则定义，`系统预警` 负责规则命中后的实例落地，两个模块不得混表。
- 所有预警实例都必须带 `rule_code` 和 `rule_version`，保证历史事件可回放。

### 4.2 统一处置状态机

- 预警状态建议统一为 `PENDING/PROCESSING/CONFIRMED/CLOSED/IGNORED`。
- 处置动作必须落 `biz_alert_handle_log`，不能只改主表状态。

## 5. 验收标准

- 场地、项目、人员、车辆、合同 5 类预警都可按真实数据查询。
- 预警详情可查看来源规则、关联对象、发生时间和处置记录。
- `AlertsMonitor.tsx` 完成主要页签的真实接口切换。
