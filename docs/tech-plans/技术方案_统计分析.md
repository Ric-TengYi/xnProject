# 技术方案 - 统计分析模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`统计分析`
- 本次覆盖的未实现功能共 `4` 个：
  - 总体分析
  - 运力分析
  - 消纳场地报表
  - 项目报表

当前仓库现状：

- 仓库已有 `DashboardProjects.tsx`、`DashboardSites.tsx`、`MonthlyReport.tsx` 等可视化骨架，但没有统一统计分析域。
- 后端未发现分析快照、聚合接口与导出任务实现。
- 统计分析应作为公共聚合层，为项目、场地、车辆和看板提供一致的数据口径。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_analysis_snapshot` | 总体分析快照 |
| `biz_transport_capacity_snapshot` | 运力分析快照 |
| `biz_project_report_snapshot` | 项目统计快照 |
| `biz_site_report_snapshot` | 场地统计快照 |
| `biz_report_export_task` | 统计报表导出任务 |

### 2.2 路由与接口分组

- `/api/analysis/overview`
- `/api/analysis/capacity`
- `/api/analysis/sites`
- `/api/analysis/projects`
- `/api/analysis/export-tasks`

### 2.3 实施顺序

1. 先统一快照层和导出任务模型。
2. 再补总体分析与运力分析。
3. 最后落地场地报表与项目报表。

---

## 3. 分功能技术方案

### 3.1 总体分析

#### 1. 数据库设计

- 新增 `biz_analysis_snapshot`，按天沉淀全局项目数、场地数、车辆数、日方量、异常数、结算量等指标。
- 索引建议：`idx_analysis_snapshot_date(tenant_id, stat_date, stat_type)`。

#### 2. 接口定义

- `GET /api/analysis/overview/summary`。
- `GET /api/analysis/overview/trend`。

#### 3. 业务逻辑

- 总体分析只做全局聚合，不直接返回明细；明细跳转通过项目/场地/车辆报表提供。
- 快照优先定时生成，页面读取缺失日期时再按需补算。

#### 4. 前端实现要点

- 可新增独立统计分析页，也可先作为 Dashboard 的深度分析入口。
- 前端图表要支持时间粒度切换和一键导出。

### 3.2 运力分析

#### 1. 数据库设计

- 新增 `biz_transport_capacity_snapshot`，沉淀运输单位、车队、车型、活跃率、故障率、违规率等指标。
- 字段建议包含 `carrier_org_id/fleet_id/vehicle_type/total_vehicle_count/active_vehicle_count/warning_count/score`。

#### 2. 接口定义

- `GET /api/analysis/capacity/ranking`。
- `GET /api/analysis/capacity/trend`。
- `POST /api/analysis/capacity/export`。

#### 3. 业务逻辑

- 运力分析要同时支持按运输单位、车队和车型三个维度切换。
- 评分模型尽量复用现有预警和维修数据，不再重复统计。

#### 4. 前端实现要点

- 前端可复用 `VehiclesManagement.tsx` 中“承运单位运力”视觉样式。
- 支持排行、趋势和明细钻取。

### 3.3 消纳场地报表

#### 1. 数据库设计

- 复用 `biz_site_report_snapshot`，沉淀场地容量利用率、异常处置、日均接车量、结算量等指标。
- 导出大表时复用 `biz_report_export_task` 异步处理。

#### 2. 接口定义

- `GET /api/analysis/sites/summary`。
- `GET /api/analysis/sites/ranking`。
- `POST /api/analysis/sites/export`。

#### 3. 业务逻辑

- 统计口径与消纳场地模块的运营报表保持一致，只是查询维度更全。
- 场地报表支持按类型、状态、运营单位过滤。

#### 4. 前端实现要点

- 复用 `DashboardSites.tsx` 的可视化卡片样式做排行展示。
- 导出中前端轮询任务状态，完成后下载。

### 3.4 项目报表

#### 1. 数据库设计

- 复用 `biz_project_report_snapshot`，沉淀项目进度、方量、交款、违规数和处置效率等指标。
- 索引建议：`idx_project_report_analysis(tenant_id, stat_date, project_id)`。

#### 2. 接口定义

- `GET /api/analysis/projects/summary`。
- `GET /api/analysis/projects/ranking`。
- `POST /api/analysis/projects/export`。

#### 3. 业务逻辑

- 项目报表与项目管理模块中的“报表统计”共用快照，避免双份实现。
- 支持按建设单位、运输单位、项目状态等维度交叉筛选。

#### 4. 前端实现要点

- 前端可复用 `DashboardProjects.tsx` 的卡片布局。
- 点击项目行时直接跳到项目详情或项目日报。

## 4. 跨功能公共约束

### 4.1 统一快照层

- 总体分析、运力分析、项目报表、场地报表都走统一快照层。
- 导出能力统一收口到 `biz_report_export_task`，避免各报表重复造轮子。

### 4.2 口径一致性

- 项目、场地、车辆统计口径必须与各业务模块展示口径一致。
- 若快照与实时数据出现偏差，应优先暴露数据更新时间与重算入口。

## 5. 验收标准

- 总体分析、运力分析、场地报表、项目报表都能真实返回聚合数据。
- 所有分析页面支持时间筛选和异步导出。
