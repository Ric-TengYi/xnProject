# 技术方案 - 安全管理模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`安全管理`
- 本次覆盖的未实现功能共 `3` 个：
  - 日常人员安全
  - 车辆运营安全
  - 消纳场地安全

当前仓库现状：

- 前端 `SecurityLedger.tsx` 已展示人员、车辆、场地三类安全检查 UI，但后端没有安全台账实体和整改流。
- 安全模块与场地、车辆、人员三条线交叉，适合做统一检查/隐患/整改模型，再按对象类型区分。
- 后续消息推送、预警配置、统计分析都要复用安全检查结果。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_security_inspection` | 安全检查记录 |
| `biz_security_issue` | 隐患问题项 |
| `biz_security_rectify` | 整改闭环记录 |
| `biz_security_plan` | 检查计划和周期规则 |

### 2.2 路由与接口分组

- `/api/security/inspections`
- `/api/security/issues`
- `/api/security/rectifies`
- `/api/security/plans`

### 2.3 实施顺序

1. 先统一安全检查、隐患、整改三张主表。
2. 再按人员、车辆、场地三个对象类型落细分能力。
3. 最后补统计、提醒和闭环看板。

---

## 3. 分功能技术方案

### 3.1 日常人员安全

#### 1. 数据库设计

- 在 `biz_security_inspection` 中以 `object_type=PERSON` 存人员检查，问题项落 `biz_security_issue`。
- 字段建议包含 `target_user_id/inspection_type/check_time/result_level/issue_count`。

#### 2. 接口定义

- `GET /api/security/inspections?objectType=PERSON`。
- `POST /api/security/inspections`。
- `POST /api/security/issues/{id}/rectify`。

#### 3. 业务逻辑

- 人员安全重点覆盖劳保穿戴、证件有效性、教育培训和现场行为。
- 不合格项进入整改流，并可触发消息提醒。

#### 4. 前端实现要点

- 把 `SecurityLedger.tsx` 的“人员日常安全”页签切到真实列表。
- 详情页展示检查记录、问题项和整改时间线。

### 3.2 车辆运营安全

#### 1. 数据库设计

- 以 `object_type=VEHICLE` 记录车辆安全检查，关联车辆主档、违规事件和维修状态。
- 问题项建议保留 `related_vehicle_id/risk_type/severity/source_event_id`。

#### 2. 接口定义

- `GET /api/security/inspections?objectType=VEHICLE`。
- `GET /api/security/issues?objectType=VEHICLE`。

#### 3. 业务逻辑

- 车辆安全检查与维修、保险、违规处置强关联，尽量共用车辆主键和事件引用。
- 若发现高风险问题，可直接联动车辆禁用或消息推送。

#### 4. 前端实现要点

- 在 `SecurityLedger.tsx` 的“车辆运营安全”页签展示真实记录。
- 列表支持按车牌、风险等级和整改状态过滤。

### 3.3 消纳场地安全

#### 1. 数据库设计

- 以 `object_type=SITE` 记录场地检查，支持消防、坝体、设备、环境等检查项。
- 字段建议包含 `site_id/check_scene/result_level/issue_count/next_check_time`。

#### 2. 接口定义

- `GET /api/security/inspections?objectType=SITE`。
- `POST /api/security/plans`：配置场地巡检计划。

#### 3. 业务逻辑

- 场地安全检查要能绑定坝体配置、设备台账和责任人，方便闭环。
- 整改完成后保留验收人和验收时间。

#### 4. 前端实现要点

- 将 `SecurityLedger.tsx` 现有时间线切到真实数据。
- 支持从场地详情页带参跳转并默认过滤当前场地。

## 4. 跨功能公共约束

### 4.1 统一对象模型

- 安全管理统一使用 `object_type + object_id` 标识检查对象。
- 问题项和整改记录都不能脱离检查记录单独存在。

### 4.2 闭环要求

- 每条隐患必须有状态、责任人、整改期限和验收结果。
- 高风险隐患应能直接触发预警与消息推送。

## 5. 验收标准

- 人员、车辆、场地三类安全检查都支持录入、查询和整改闭环。
- `SecurityLedger.tsx` 三个页签均切到真实数据。
