## 3. 当前方案说明

### 3.1 本轮无新增开发项

#### 1. 数据库设计

- 不新增表。
- 继续复用 `sys_role`、`sys_role_permission_rel`、`sys_data_scope_rule`。

#### 2. 接口定义

- 不新增模块级接口。
- 继续沿用现有角色列表、详情、权限配置和数据范围配置接口完成联调。

#### 3. 业务逻辑

- 角色模型与用户体系拆分文档保持一致，不在当前阶段调整编码、作用域和主键关系。
- 角色管理的后续工作重点应是验收现有逻辑是否稳定，而不是继续扩张字段或表结构。

#### 4. 前端实现要点

- `RolesManagement.tsx` 继续使用现有接口完成联调。
- 若出现体验问题，优先补页面交互和权限树展示，不改后端主模型。

### 3.2 后续可选增强（不属于本轮未实现功能）

#### 1. 数据库设计

- 如未来需要按钮权限分组或权限模板，可在 `sys_permission` 上补 `group_code`、`sort_order` 等字段。

#### 2. 接口定义

- 可选补 `GET /api/roles/{id}/menu-tree`、`GET /api/permissions` 等增强接口统一前端权限树体验。

#### 3. 业务逻辑

- 前端当前保存权限时仍以菜单勾选为主，后续可再补按钮权限和接口权限联动预览。

#### 4. 前端实现要点

- 可在右侧配置区增加按钮权限、接口权限和数据范围预览，但应作为二期增强。

## 4. 跨功能公共约束

### 4.1 不重复造轮子

- `角色管理` 与 `菜单/权限点/数据权限` 已存在清晰边界，本轮不建议重新设计角色主模型。
- 若后续出现需求扩展，应优先复用用户体系文档已冻结的编码和接口路径。

## 5. 验收标准

- 角色列表、角色详情、角色新增可正常联调。
- 角色权限与数据范围接口可稳定返回与保存。
- 本模块不再新增“未实现功能”开发项。
