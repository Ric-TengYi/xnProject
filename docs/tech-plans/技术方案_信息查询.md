# 技术方案 - 信息查询模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`信息查询`
- 本次覆盖的未实现功能共 `1` 个：
  - 打卡数据

当前仓库现状：

- 仓库未发现专门的信息查询页面和后端接口，打卡数据目前只在若干业务页面文案中被引用。
- 后续小程序、项目打卡配置、违规研判都依赖统一的打卡记录事实表。
- 虽然本模块只剩 1 个功能，但建议按独立查询域建设，避免把打卡查询散落在多个模块里。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_checkin_record` | 打卡记录事实表 |
| `biz_checkin_attachment` | 打卡照片与附件 |
| `biz_checkin_abnormal` | 打卡异常标记和人工复核结果 |

### 2.2 路由与接口分组

- `/api/checkins`
- `/api/checkins/summary`

### 2.3 实施顺序

1. 先建打卡记录事实表和统一查询接口。
2. 再补照片、异常标记与详情能力。

---

## 3. 分功能技术方案

### 3.1 打卡数据

#### 1. 数据库设计

- 新增 `biz_checkin_record`，字段建议包含 `tenant_id/project_id/site_id/vehicle_id/user_id/checkin_type/checkin_time/longitude/latitude/source_type/checkin_status`。
- 附件单独存在 `biz_checkin_attachment`，异常复核落 `biz_checkin_abnormal`。

#### 2. 接口定义

- `GET /api/checkins`：按项目、场地、车辆、人员、时间分页查询。
- `GET /api/checkins/{id}`：打卡详情。
- `GET /api/checkins/summary`：打卡汇总统计。

#### 3. 业务逻辑

- 打卡记录统一承接 Web、设备、小程序三类来源，禁止多个模块分别建表。
- 查询接口同时返回命中规则、位置判断结果和异常标签，方便后续研判复用。

#### 4. 前端实现要点

- 可先新增一个独立信息查询页，也可先挂到 `Placeholder.tsx` 路由承载。
- 详情页要支持查看位置点、打卡图片和异常复核结果。

## 4. 跨功能公共约束

### 4.1 事实表优先

- 打卡数据必须先形成统一事实表，再衍生日报、预警和小程序申报。
- 所有上游来源都写同一主键体系，不允许新增并行“导入表”“临时表”。

## 5. 验收标准

- 支持按项目、场地、车辆、人员和时间查询打卡数据。
- 打卡详情可查看图片、位置、异常标签和命中规则。
