# 技术方案 - 事件管理模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`事件管理`
- 本次覆盖的未实现功能共 `2` 个：
  - 人工事件
  - 事件审核

当前仓库现状：

- 后端只有 `AlertsController` 的空列表接口和 `biz_alert_event` 简单事件表骨架，无法承接人工事件的建单、流转、审核和闭环。
- 审批规则能力已经在 `sys_approval_actor_rule` 中具备基础模型，但尚未接入“事件审核”业务。
- 前端 `EventsManagement.tsx` 仍使用本地 mock 数据，审核按钮、详情抽屉和状态流转都未落后端。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_manual_event` | 人工事件主表 |
| `biz_manual_event_attachment` | 事件附件与图片 |
| `biz_manual_event_audit_log` | 事件审核与流转记录 |
| `biz_manual_event_handle_log` | 事件处置记录 |

### 2.2 路由与接口分组

- `/api/events`
- `/api/events/{id}`
- `/api/events/{id}/submit`
- `/api/events/{id}/approve`
- `/api/events/{id}/reject`
- `/api/events/{id}/close`
- `/api/events/pending-audits`

### 2.3 实施顺序

1. 先落人工事件主表与列表、详情、创建能力。
2. 再接审批规则，打通审核流转。
3. 最后补处置记录、附件和关闭闭环。

---

## 3. 分功能技术方案

### 3.1 人工事件

#### 1. 数据库设计

- 新增 `biz_manual_event`，核心字段建议为 `tenant_id/event_no/event_type/title/content/source_channel/project_id/site_id/vehicle_id/reporter_id/reporter_name/priority/status/current_audit_node/occur_time/report_time`。
- 新增 `biz_manual_event_attachment`，保存 `event_id/file_name/file_url/file_type/file_size`。
- 索引建议：
  - `uk_manual_event_no(tenant_id, event_no)`
  - `idx_manual_event_status_time(tenant_id, status, report_time)`
  - `idx_manual_event_target(tenant_id, project_id, site_id, vehicle_id)`

#### 2. 接口定义

- `GET /api/events`：按事件编号、类型、状态、项目、场地、时间分页查询。
- `POST /api/events`：创建人工事件。
- `GET /api/events/{id}`：查看事件详情。
- `POST /api/events/{id}/submit`：提交审核。
- `POST /api/events/{id}/close`：关闭事件。

#### 3. 业务逻辑

- 人工事件与系统预警要分开建模，人工事件强调“人工上报与人工处理”，不应直接复用预警流水表。
- 事件状态建议统一为 `DRAFT/PENDING_AUDIT/PROCESSING/CLOSED/REJECTED`。
- 事件创建时可关联项目、场地、车辆等业务对象，为后续跳转与统计提供稳定主键。

#### 4. 前端实现要点

- `EventsManagement.tsx` 改为真实列表查询，并新增“新建事件”弹窗或抽屉。
- 详情抽屉需要展示基础信息、描述、附件、审核记录和处置记录。
- 列表页支持按状态和事件类型切换，并能从项目、场地、预警详情页带参跳转。

### 3.2 事件审核

#### 1. 数据库设计

- 新增 `biz_manual_event_audit_log`，字段建议为 `event_id/node_code/action/result/auditor_id/auditor_name/comment/audit_time`。
- 新增 `biz_manual_event_handle_log`，记录 `event_id/handler_id/handle_action/handle_result/remark/handle_time`。
- 如需指派处理人，可在主表补 `assignee_id/assignee_name` 字段。

#### 2. 接口定义

- `GET /api/events/pending-audits`：查询待审核事件。
- `POST /api/events/{id}/approve`：审核通过。
- `POST /api/events/{id}/reject`：审核驳回。
- `POST /api/events/{id}/assign`：指派处理人。

#### 3. 业务逻辑

- 审核人解析复用 `sys_approval_actor_rule`，建议业务类型固定为 `MANUAL_EVENT_AUDIT`。
- 审核通过后，事件状态切换为 `PROCESSING`，并生成首条处置记录。
- 驳回必须保留审核意见，后续允许发起人修改后再次提交，避免一票否决后只能重建事件。

#### 4. 前端实现要点

- `EventsManagement.tsx` 列表中的“审核/详情”动作改为真实接口驱动。
- 事件详情页增加审核时间轴和审批意见区。
- 待审核状态下展示“通过/退回”操作，处理中状态下展示“关闭/转派”操作。

## 4. 跨功能公共约束

### 4.1 与预警边界

- 系统预警负责机器触发的告警，事件管理负责人为主导的人工建单和闭环，两个域可以互相关联但不能混表。
- 如人工事件来自某条预警，建议在 `biz_manual_event` 中保留 `source_alert_id`，而不是复制整条预警数据。

### 4.2 审批与审计

- 事件审核、转派、关闭都必须留痕，不允许只改主表状态。
- 附件、意见、处理记录要可回溯，满足后续审计和复盘需要。

## 5. 验收标准

- 人工事件支持创建、列表、详情、提交审核和关闭。
- 事件审核支持待审查询、通过、驳回和意见留痕。
- `EventsManagement.tsx` 完成从 mock 到真实接口切换。
