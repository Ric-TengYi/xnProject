# 技术方案 - 组织管理模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`组织管理`
- 本次覆盖的未实现功能共 `1` 个：
  - 组织类型

当前仓库现状：

- 后端 `OrgsController` 已实现组织树、组织详情、组织新增、负责人变更、状态变更和删除。
- `sys_org.org_type` 目前只是普通字符串字段，仓库内未发现专门的组织类型主数据表或管理接口。
- 前端 `Organization.tsx` 已有组织树与人员列表骨架，但没有组织类型配置入口，组织新增仍缺少可维护的类型字典。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `sys_org` | 继续作为组织树主表，`org_type` 改为引用组织类型编码 |
| `sys_org_type` | 组织类型字典表，维护类型编码、名称、排序、适用范围、状态 |

`sys_org_type` 建议字段：

- `type_code`
- `type_name`
- `tenant_id`（为空表示平台级）
- `status`
- `sort_order`
- `allow_child_flag`
- `default_flag`
- `remark`

### 2.2 路由与接口分组

- `/api/orgs`
- `/api/orgs/tree`
- `/api/org-types`
- `/api/org-types/{id}`
- `/api/org-types/options`

### 2.3 实施顺序

1. 先新增 `sys_org_type`，冻结类型编码。
2. 再把组织新增/编辑接口改为只接收合法类型编码。
3. 最后补组织类型维护页面和组织树筛选。

---

## 3. 分功能技术方案

### 3.1 组织类型

#### 1. 数据库设计

- 新增 `sys_org_type`，核心字段建议为 `id/type_code/type_name/tenant_id/status/sort_order/allow_child_flag/default_flag/remark/create_time/update_time/deleted`。
- `sys_org.org_type` 保持字符串编码，但增加外键语义校验，避免前端随意传值。
- 索引建议：
  - `uk_sys_org_type_scope_code(tenant_id, type_code)`
  - `idx_sys_org_type_scope_status(tenant_id, status, sort_order)`

#### 2. 接口定义

- `GET /api/org-types`：分页查询组织类型。
- `GET /api/org-types/options`：返回可用于下拉的启用类型。
- `POST /api/org-types`：新增组织类型。
- `PUT /api/org-types/{id}`：编辑组织类型。
- `PUT /api/org-types/{id}/status`：启停用组织类型。

#### 3. 业务逻辑

- 组织类型应优先支持平台默认 + 租户自定义两层作用域，避免每个租户都复制一份平台类型。
- 已被组织引用的类型不应直接物理删除，建议仅停用并保留历史可回放能力。
- 组织新增、编辑时只允许使用“当前租户可见且状态为启用”的组织类型。

#### 4. 前端实现要点

- 在 `Organization.tsx` 中新增“组织类型管理”抽屉或弹窗。
- 组织树右侧的新增/编辑表单改为从 `GET /api/org-types/options` 拉取类型下拉。
- 组织类型列表建议支持“平台默认”“租户自定义”“已停用”三种筛选。

## 4. 跨功能公共约束

### 4.1 类型编码冻结

- 所有组织类型都必须使用稳定编码，不能让前端直接提交中文名称作为 `org_type`。
- 后续数据权限、审批参与人规则若需要按组织类型选人，必须复用同一套类型编码。

### 4.2 与其他模块依赖

- `组织人员管理` 需要通过组织类型做筛选与详情展示。
- `审核审批配置` 可按组织类型解析负责人或审批链。

## 5. 验收标准

- 组织类型支持列表、启停用、下拉选项查询。
- 组织新增和编辑只能选择合法的组织类型编码。
- 已引用类型停用后不影响历史组织树展示。
