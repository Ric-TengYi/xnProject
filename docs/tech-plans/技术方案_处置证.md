# 技术方案 - 处置证模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`处置证`
- 本次覆盖的未实现功能共 `3` 个：
  - 处置证同步
  - 处置证新增
  - 处置证关联

当前仓库现状：

- 仓库中未发现专门的处置证 controller、service、entity 或数据库表。
- 前端只有 `ProjectsPermits.tsx` 提供了项目侧“处置证清单”的 mock 页面，尚无独立模块入口。
- 处置证天然连接项目、场地、车辆和合同，建议先冻结主档模型，再实现同步、人工新增和关联。

## 2. 模块总设计

### 2.1 核心对象

| 表名 | 用途 |
| --- | --- |
| `biz_disposal_permit` | 处置证主表，保存证号、证型、来源、状态、有效期、签发信息 |
| `biz_disposal_permit_bind` | 处置证与项目/场地/车辆/合同的关联表 |
| `biz_permit_sync_task` | 处置证同步任务主表 |
| `biz_permit_sync_log` | 同步批次和失败明细 |
| `biz_disposal_permit_attachment` | 证件影像与附件 |

### 2.2 路由与接口分组

- `/api/permits`
- `/api/permits/sync`
- `/api/permits/bindings`
- `/api/permits/import-*`

### 2.3 实施顺序

1. 先建 `biz_disposal_permit` 与关联表，补齐证件主档能力。
2. 再实现 `处置证同步`，优先打通外部数据导入与幂等更新。
3. 随后补 `处置证新增` 和 `处置证关联`，形成证件全生命周期。

---

## 3. 分功能技术方案

### 3.1 处置证同步

#### 1. 数据库设计

- 新增 `biz_permit_sync_task` 与 `biz_permit_sync_log`，记录外部拉取时间窗、成功数、失败数、原始报文摘要。
- `biz_disposal_permit` 需预留 `external_id/source_system/source_hash/last_sync_time` 字段做幂等。

#### 2. 接口定义

- `POST /api/permits/sync`：手动触发同步。
- `GET /api/permits/sync/tasks`：查询同步任务。
- `GET /api/permits/sync/tasks/{id}`：查看失败明细。

#### 3. 业务逻辑

- 同步按 `external_id + tenant_id` 幂等更新，避免重复落证。
- 对已过期、已作废和状态冲突的证件要保留同步告警，不直接静默覆盖。

#### 4. 前端实现要点

- 模块首页增加“同步记录”和“立即同步”入口。
- 同步结果需展示新增、更新、失败三类统计，并支持下载错误清单。

### 3.2 处置证新增

#### 1. 数据库设计

- 在 `biz_disposal_permit` 中补齐 `permit_no/permit_type/issue_org/issue_date/effective_date/expire_date/status` 等核心字段。
- 附件通过 `biz_disposal_permit_attachment` 单独管理，证件表只存主图或封面标记。

#### 2. 接口定义

- `POST /api/permits`：新增处置证。
- `PUT /api/permits/{id}`：编辑证件。
- `GET /api/permits/{id}`：查看详情。

#### 3. 业务逻辑

- 人工新增证件时需要校验证号唯一、有效期合法、关联租户正确。
- 证件作废不物理删除，只改状态并保留作废原因。

#### 4. 前端实现要点

- 新增独立的处置证表单页或抽屉，支持附件上传与OCR后补录。
- 从项目、场地、车辆入口发起新增时，允许自动带入上下文信息。

### 3.3 处置证关联

#### 1. 数据库设计

- 新增 `biz_disposal_permit_bind`，字段建议为 `permit_id/biz_type/biz_id/site_id/vehicle_id/contract_id/bind_status/effective_time/expire_time`。
- 索引建议：`idx_permit_bind_biz(tenant_id, biz_type, biz_id, bind_status)`。

#### 2. 接口定义

- `POST /api/permits/{id}/bindings`：建立关联。
- `GET /api/permits/{id}/bindings`：查询当前与历史关系。
- `DELETE /api/permits/bindings/{bindingId}`：解除绑定。

#### 3. 业务逻辑

- 一张处置证可关联多个业务对象，但同一时间段内对同一车辆只允许存在一条有效关系。
- 解除绑定后保留历史，不直接删除关系记录。

#### 4. 前端实现要点

- 项目详情、场地详情和车辆详情都复用同一关联弹窗。
- 关系清单需突出展示“当前有效 / 即将过期 / 已解绑”三种状态。

## 4. 跨功能公共约束

### 4.1 证件状态机

- 建议统一使用 `DRAFT/VALID/EXPIRING/EXPIRED/VOID` 状态。
- 证件状态由同步任务、人工编辑和定时过期任务共同维护，但只能通过统一 service 改写。

### 4.2 业务依赖

- 项目管理、车辆管理、消纳场地、小程序都可能读取处置证有效性。
- 处置证关联接口必须提供统一的校验结果，避免各模块自行判断是否可用。

## 5. 验收标准

- 处置证支持同步、人工新增、关联解绑三类完整操作。
- 项目、场地、车辆均可读取同一张证件详情和关联历史。
- 同步任务具备幂等、失败明细和错误回溯能力。
