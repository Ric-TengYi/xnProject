# 技术方案 - 合同结算模块

## 1. 范围与现状

- 来源：`功能实现状态分析.xlsx`
- 模块：`合同结算`
- 本次覆盖的未实现功能共 `10` 个：
  - 合同入账
  - 合同导出
  - 合同导入
  - 月报统计
  - 单位统计
  - 变更合同发起
  - 消纳延期
  - 内拨申请
  - 项目结算
  - 场地结算

当前仓库现状：

- 后端仅有 `biz_contract` 基础表和 `ContractsController` 占位接口。
- 前端已有 `ContractsManagement.tsx`、`ContractsPayments.tsx`、`Settlements.tsx`、`MonthlyReport.tsx`、`ContractDetail.tsx` 页面骨架，均为 mock 数据。
- 审批配置已具备基础规则能力，可复用 `sys_approval_actor_rule`。

## 2. 模块总设计

### 2.1 核心对象

建议先冻结合同结算域的 8 张核心表：

| 表名 | 用途 |
| --- | --- |
| `biz_contract` | 合同主表 |
| `biz_contract_attachment` | 合同及申请附件 |
| `biz_contract_receipt` | 合同入账记录 |
| `biz_contract_change_apply` | 合同变更申请 |
| `biz_contract_extension_apply` | 延期申请 |
| `biz_contract_transfer_apply` | 内拨申请 |
| `biz_settlement_order` | 结算单主表 |
| `biz_settlement_item` | 结算明细 |

补充支撑表：

| 表名 | 用途 |
| --- | --- |
| `biz_contract_import_batch` | 导入批次 |
| `biz_contract_import_error` | 导入错误明细 |
| `biz_report_export_task` | 异步导出任务 |
| `biz_contract_stat_snapshot` | 月报/单位统计快照 |

### 2.2 主表升级建议

现有 `biz_contract` 字段不足以支撑结算，建议补齐：

| 字段 | 说明 |
| --- | --- |
| `tenant_id` | 租户隔离 |
| `contract_no` | 合同编号，唯一 |
| `contract_type` | 合同类型 |
| `project_id` | 关联项目 |
| `site_id` | 约定场地 |
| `construction_org_id` | 建设单位 |
| `transport_org_id` | 运输单位 |
| `site_operator_org_id` | 场地方/运营方 |
| `sign_date` | 签订日期 |
| `effective_date` | 生效日期 |
| `expire_date` | 到期日期 |
| `agreed_volume` | 约定方量 |
| `unit_price` | 单价 |
| `contract_amount` | 合同总金额 |
| `received_amount` | 累计已入账金额 |
| `settled_amount` | 累计已结算金额 |
| `change_version` | 变更版本号 |
| `approval_status` | 审批状态 |

建议索引：

- `uk_contract_no(tenant_id, contract_no)`
- `idx_contract_project_status(tenant_id, project_id, contract_status)`
- `idx_contract_site_status(tenant_id, site_id, contract_status)`

- `idx_contract_expire_date(tenant_id, expire_date, contract_status)`

### 2.3 路由与接口分组

建议统一在以下资源下扩展：

- `/api/contracts`
- `/api/contracts/receipts`
- `/api/contracts/import-*`
- `/api/contracts/export`
- `/api/contracts/change-applications`
- `/api/contracts/extensions`
- `/api/contracts/transfers`
- `/api/settlements`
- `/api/reports/contracts/*`
- `/api/export-tasks/*`

### 2.4 实施顺序

建议按下面顺序落地，能最快形成可联调闭环：

1. 升级 `biz_contract` + 新增 `biz_contract_receipt`
2. 打通 `合同入账`
3. 打通 `合同导入 / 导出`
4. 打通 `变更 / 延期 / 内拨` 审批链
5. 打通 `项目结算 / 场地结算`
6. 最后补 `月报统计 / 单位统计`

## 3. 分功能技术方案

### 3.1 合同入账

#### 1. 数据库设计

- 升级 `biz_contract`
  - 增加 `received_amount`、`approval_status`、`contract_status`、`tenant_id`
- 新增 `biz_contract_receipt`
  - 核心字段：`id`、`tenant_id`、`contract_id`、`receipt_no`、`receipt_date`、`amount`、`receipt_type`、`voucher_no`、`bank_flow_no`、`status`、`operator_id`、`remark`、`create_time`
- 新增 `biz_contract_attachment`
  - 核心字段：`id`、`tenant_id`、`biz_type`、`biz_id`、`file_name`、`file_url`、`file_size`、`mime_type`
- 索引
  - `uk_receipt_no(tenant_id, receipt_no)`
  - `idx_receipt_contract_date(tenant_id, contract_id, receipt_date)`
  - `idx_receipt_status_date(tenant_id, status, receipt_date)`

#### 2. 接口定义

- `GET /api/contracts/receipts`
  - 查询：`contractId`、`keyword`、`status`、`startDate`、`endDate`、`pageNo`、`pageSize`
- `POST /api/contracts/{id}/receipts`
  - 请求体：`amount`、`receiptDate`、`voucherNo`、`bankFlowNo`、`attachmentIds`、`remark`
- `GET /api/contracts/{id}/receipts`
  - 返回该合同全部入账流水
- `GET /api/contracts/receipts/{receiptId}`
  - 返回单笔入账详情
- `PUT /api/contracts/receipts/{receiptId}/cancel`
  - 用于冲销错误入账

#### 3. 业务逻辑

- 仅 `生效/执行中` 合同允许登记入账。
- 单笔入账金额必须大于 `0`，累计入账金额默认不能超过合同总金额；若允许超额，需走审批开关。
- 创建入账记录与回写 `biz_contract.received_amount` 必须同事务完成。
- 冲销不物理删除，改为写状态 `CANCELLED` 并生成反向流水，保证审计链完整。
- 合同详情页的“资金流向”直接读取合同汇总 + 最近入账记录，不再由前端计算。

#### 4. 前端实现要点

- 复用 `xngl-web/src/pages/ContractsPayments.tsx` 作为主页面。
- 列表页改为调用 `GET /api/contracts/receipts`，弹窗提交改为 `POST /api/contracts/{id}/receipts`。
- 上传凭证走附件接口，提交时只传 `attachmentIds`。
- `ContractDetail.tsx` 的“入账与发票”页签改为真实数据，支持查看冲销状态和附件预览。

### 3.2 合同导出

#### 1. 数据库设计

- 新增 `biz_report_export_task`
  - 核心字段：`id`、`tenant_id`、`biz_type`、`export_type`、`query_json`、`file_name`、`file_url`、`status`、`fail_reason`、`creator_id`、`expire_time`
- 索引
  - `idx_export_task_biz_status(tenant_id, biz_type, status, create_time)`
  - `idx_export_task_creator(tenant_id, creator_id, create_time)`

#### 2. 接口定义

- `POST /api/contracts/export`
  - 请求体：列表筛选条件 + `exportType`
- `GET /api/export-tasks/{id}`
  - 查询任务状态
- `GET /api/export-tasks/{id}/download`
  - 下载导出文件

#### 3. 业务逻辑

- 导出默认走异步任务，避免大数据量阻塞请求线程。
- 导出字段与合同列表保持一致，附加 `累计入账`、`待入账`、`结算状态`、`审批状态`。
- 导出查询必须复用列表查询条件，保证“所见即所得”。
- 对已过期文件定时清理，避免本地或对象存储长期堆积。

#### 4. 前端实现要点

- `ContractsManagement.tsx` 的“导出月度报表”按钮拆成“导出合同清单”和“导出月报”两类动作。
- 前端提交当前筛选条件后轮询导出任务状态，完成后自动下载。
- 导出中按钮置灰，避免重复点击产生多批任务。

### 3.3 合同导入

#### 1. 数据库设计

- 新增 `biz_contract_import_batch`
  - 核心字段：`id`、`tenant_id`、`batch_no`、`file_name`、`file_url`、`total_count`、`success_count`、`fail_count`、`status`、`operator_id`
- 新增 `biz_contract_import_error`
  - 核心字段：`id`、`batch_id`、`row_no`、`contract_no`、`error_code`、`error_message`、`raw_json`
- 复用 `biz_contract`
- 索引
  - `uk_import_batch_no(tenant_id, batch_no)`
  - `idx_import_batch_status(tenant_id, status, create_time)`
  - `idx_import_error_batch(batch_id, row_no)`

#### 2. 接口定义

- `POST /api/contracts/import-preview`
  - 上传模板后返回预校验结果
- `POST /api/contracts/import-commit`
  - 请求体：`batchId`
- `GET /api/contracts/import-batches`
  - 查询导入批次
- `GET /api/contracts/import-batches/{id}`
  - 查询批次详情与错误列表
- `GET /api/contracts/import-template`
  - 下载导入模板

#### 3. 业务逻辑

- 导入分 `预校验` 和 `正式提交` 两步，避免用户直接落脏数据。
- 预校验需校验：
  - 合同编号是否重复
  - 项目、场地、建设单位、运输单位是否存在
  - 金额、方量、日期格式是否合法
- 正式提交按行独立处理，成功与失败分开统计，不因一条错误整批回滚。
- 同一模板文件可通过文件哈希做幂等保护，避免重复导入。

#### 4. 前端实现要点

- 在 `ContractsManagement.tsx` 增加“导入合同”入口，使用三步向导：上传模板、预校验、确认导入。
- 错误行支持导出错误清单，便于业务修正后重新上传。
- 列表导入成功后自动按批次高亮新增合同。

### 3.4 月报统计

#### 1. 数据库设计

- 新增 `biz_contract_stat_snapshot`
  - 核心字段：`id`、`tenant_id`、`stat_date`、`stat_month`、`stat_type`、`dimension_type`、`dimension_id`、`contract_count`、`new_contract_count`、`contract_amount`、`receipt_amount`、`settlement_amount`、`agreed_volume`、`actual_volume`
- 统计来源
  - `biz_contract`
  - `biz_contract_receipt`
  - `biz_settlement_order`
- 索引
  - `idx_contract_stat_month(tenant_id, stat_month, stat_type)`
  - `idx_contract_stat_dimension(tenant_id, dimension_type, dimension_id, stat_month)`

#### 2. 接口定义

- `GET /api/reports/contracts/monthly/summary`
- `GET /api/reports/contracts/monthly/trend`
- `GET /api/reports/contracts/monthly/types`
- `POST /api/reports/contracts/monthly/export`

#### 3. 业务逻辑

- 月报口径统一按 `统计月份 + 租户 + 数据权限` 生成。
- 合同金额按 `sign_date` 统计新增合同，入账金额按 `receipt_date`，结算金额按 `settlement_date`。
- 报表优先读取月度快照；若快照不存在，则实时聚合并回填快照。
- 月末可由定时任务生成 `T+1` 快照，降低页面实时统计压力。

#### 4. 前端实现要点

- 复用 `xngl-web/src/pages/MonthlyReport.tsx`。
- 页面拆成三个请求块：顶部汇总、趋势图、类型明细。
- “导出月报”直接走导出任务接口，不在浏览器本地拼装 Excel。
- 统计月份切换时保留上次筛选条件，避免频繁重置。

### 3.5 单位统计

#### 1. 数据库设计

- 复用 `biz_contract_stat_snapshot`
- 建议在 `biz_contract` 固化组织维度字段：
  - `construction_org_id`
  - `transport_org_id`
  - `site_operator_org_id`
- 若业务要求按单位排名长期查询，可增加快照维度 `dimension_type=ORG`
- 索引
  - `idx_contract_org_dimension(tenant_id, construction_org_id, sign_date)`
  - `idx_contract_transport_dimension(tenant_id, transport_org_id, sign_date)`

#### 2. 接口定义

- `GET /api/reports/contracts/unit-stats`
  - 查询：`unitType`、`month`、`keyword`、`pageNo`、`pageSize`
- `GET /api/reports/contracts/unit-stats/{orgId}/trend`
  - 查看单位趋势

#### 3. 业务逻辑

- 单位统计至少支持三种口径：建设单位、运输单位、场地运营单位。
- 输出指标建议固定为：
  - 合同数
  - 合同总金额
  - 累计入账
  - 待入账
  - 已结算金额
  - 实际消纳方量
- 与月报共用快照计算逻辑，仅切换统计维度，避免重复开发两套聚合。

#### 4. 前端实现要点

- 建议作为 `MonthlyReport.tsx` 的二级页签，避免新增独立页面。
- 支持按单位类型切换和 Top N 排名展示。
- 单位行点击后进入趋势抽屉或右侧详情面板，不必新开路由。

### 3.6 变更合同发起

#### 1. 数据库设计

- 新增 `biz_contract_change_apply`
  - 核心字段：`id`、`tenant_id`、`change_no`、`contract_id`、`change_type`、`before_snapshot_json`、`after_snapshot_json`、`reason`、`approval_status`、`process_instance_id`、`current_node_code`、`applicant_id`
- 可选新增 `biz_contract_change_item`
  - 核心字段：`apply_id`、`field_code`、`before_value`、`after_value`
- 索引
  - `uk_change_no(tenant_id, change_no)`
  - `idx_change_contract_status(tenant_id, contract_id, approval_status, create_time)`

#### 2. 接口定义

- `POST /api/contracts/{id}/change-applications`
- `GET /api/contracts/{id}/change-applications`
- `GET /api/contracts/change-applications/{applyId}`
- `POST /api/contracts/change-applications/{applyId}/submit`
- `POST /api/contracts/change-applications/{applyId}/approve`
- `POST /api/contracts/change-applications/{applyId}/reject`

#### 3. 业务逻辑

- 允许变更的字段建议限定为：金额、单价、约定方量、场地、有效期、单位信息、备注。
- 发起时保存前后快照，不直接覆盖主合同。
- 审批通过后再回写 `biz_contract`，并将 `change_version + 1`。
- 审批参与人解析复用 `sys_approval_actor_rule`，`biz_type` 建议固定为 `CONTRACT_CHANGE`。

#### 4. 前端实现要点

- 复用 `ContractsManagement.tsx` 的“变更”操作入口。
- 弹窗或抽屉中同时展示“变更前/变更后”对比，重点字段高亮。
- `ContractDetail.tsx` 新增“变更记录”区域，展示审批流和版本历史。

### 3.7 消纳延期

#### 1. 数据库设计

- 新增 `biz_contract_extension_apply`
  - 核心字段：`id`、`tenant_id`、`apply_no`、`contract_id`、`original_expire_date`、`requested_expire_date`、`requested_volume_delta`、`reason`、`approval_status`、`process_instance_id`、`applicant_id`
- 复用 `biz_contract_attachment`
- 索引
  - `uk_extension_no(tenant_id, apply_no)`
  - `idx_extension_contract_status(tenant_id, contract_id, approval_status, create_time)`

#### 2. 接口定义

- `POST /api/contracts/{id}/extensions`
- `GET /api/contracts/{id}/extensions`
- `GET /api/contracts/extensions/{id}`
- `POST /api/contracts/extensions/{id}/submit`
- `POST /api/contracts/extensions/{id}/approve`
- `POST /api/contracts/extensions/{id}/reject`

#### 3. 业务逻辑

- 仅允许对 `生效中且未终止` 的合同发起延期。
- `requested_expire_date` 必须大于当前到期时间；若同时增加方量，必须记录 `requested_volume_delta`。
- 审批通过后更新 `biz_contract.expire_date`，必要时同步更新 `agreed_volume`。
- 审批类型建议使用 `biz_type=CONTRACT_EXTENSION`。

#### 4. 前端实现要点

- 在 `ContractDetail.tsx` 基础信息区增加“延期申请”按钮。
- 表单中显示当前到期时间、剩余方量、剩余金额，减少误填。
- 延期历史放在详情页时间轴，与审批状态联动。

### 3.8 内拨申请

> 说明：当前仓库没有“内拨”业务定义。此处按“同租户内合同剩余额度/方量在项目或场地之间调拨”设计；若后续业务确认仅为资金内拨，可保留表结构，只使用金额字段。

#### 1. 数据库设计

- 新增 `biz_contract_transfer_apply`
  - 核心字段：`id`、`tenant_id`、`transfer_no`、`source_contract_id`、`target_contract_id`、`transfer_amount`、`transfer_volume`、`reason`、`approval_status`、`process_instance_id`、`applicant_id`
- 可选新增 `biz_contract_transfer_detail`
  - 核心字段：`apply_id`、`source_before_json`、`target_before_json`、`source_after_json`、`target_after_json`
- 索引
  - `uk_transfer_no(tenant_id, transfer_no)`
  - `idx_transfer_source_status(tenant_id, source_contract_id, approval_status)`
  - `idx_transfer_target_status(tenant_id, target_contract_id, approval_status)`

#### 2. 接口定义

- `POST /api/contracts/transfers`
- `GET /api/contracts/transfers`
- `GET /api/contracts/transfers/{id}`
- `POST /api/contracts/transfers/{id}/submit`
- `POST /api/contracts/transfers/{id}/approve`
- `POST /api/contracts/transfers/{id}/reject`

#### 3. 业务逻辑

- 源合同与目标合同必须属于同一租户，且均为可执行状态。
- 内拨时校验源合同剩余金额/方量是否充足。
- 审批通过后，事务内同步扣减源合同额度并增加目标合同额度。
- 审批类型建议使用 `biz_type=CONTRACT_TRANSFER`，便于和变更、延期分流。

#### 4. 前端实现要点

- 建议在合同详情页增加“内拨申请”按钮，默认带出当前合同作为源合同。
- 表单中实时展示源/目标合同剩余额度，避免超拨。
- 列表页需要增加“内拨中”状态标签，便于业务追踪。

### 3.9 项目结算

#### 1. 数据库设计

- 新增 `biz_settlement_order`
  - 核心字段：`id`、`tenant_id`、`settlement_no`、`settlement_type`、`target_project_id`、`target_site_id`、`period_start`、`period_end`、`total_volume`、`unit_price`、`total_amount`、`adjust_amount`、`payable_amount`、`approval_status`、`settlement_status`、`process_instance_id`、`creator_id`
- 新增 `biz_settlement_item`
  - 核心字段：`id`、`settlement_order_id`、`source_record_type`、`source_record_id`、`project_id`、`site_id`、`vehicle_id`、`biz_date`、`volume`、`unit_price`、`amount`
- 索引
  - `uk_settlement_no(tenant_id, settlement_no)`
  - `idx_settlement_project_period(tenant_id, settlement_type, target_project_id, period_start, period_end)`
  - `idx_settlement_item_order(settlement_order_id, biz_date)`

#### 2. 接口定义

- `POST /api/settlements/project/generate`
  - 请求体：`projectId`、`periodStart`、`periodEnd`、`pricingMode`
- `GET /api/settlements`
  - 查询：`settlementType`、`status`、`projectId`、`siteId`、`pageNo`、`pageSize`
- `GET /api/settlements/{id}`
- `POST /api/settlements/{id}/submit`
- `POST /api/settlements/{id}/approve`
- `POST /api/settlements/{id}/reject`
- `POST /api/settlements/{id}/export`

#### 3. 业务逻辑

- 项目结算按“项目 + 结算周期”聚合，不允许同周期重复生成已生效结算单。
- 明细来源优先对接后续 `biz_disposal_record`；若站点模块尚未完成，一期允许通过明细导入补数。
- 结算草稿允许人工调整 `adjust_amount`，但不得直接改原始明细量。
- 审批通过后将结算单状态改为 `SETTLED`，并回写合同的 `settled_amount`。

#### 4. 前端实现要点

- 复用 `xngl-web/src/pages/Settlements.tsx`，以“按项目结算”作为一种类型视图。
- 新增“发起新结算”向导：选择项目、选择周期、预览明细、提交审批。
- 详情页需要支持汇总卡片 + 明细表格 + 审批时间轴。

### 3.10 场地结算

#### 1. 数据库设计

- 复用 `biz_settlement_order`、`biz_settlement_item`
- `settlement_type` 固定为 `SITE`
- 关键字段使用：
  - `target_site_id`
  - `target_project_id` 可为空
- 索引
  - `idx_settlement_site_period(tenant_id, settlement_type, target_site_id, period_start, period_end)`

#### 2. 接口定义

- `POST /api/settlements/site/generate`
- `GET /api/settlements?settlementType=SITE`
- `GET /api/settlements/{id}`
- `POST /api/settlements/{id}/submit`
- `POST /api/settlements/{id}/approve`
- `POST /api/settlements/{id}/reject`
- `POST /api/settlements/{id}/export`

#### 3. 业务逻辑

- 场地结算按“场地 + 结算周期”聚合，统计该周期进入场地的有效消纳记录。
- 同一张结算表同时服务两个模块入口：
  - `合同结算` 模块侧重财务回款
  - `消纳场地` 模块侧重运营对账
- 若一条消纳记录已被项目结算和场地结算同时引用，需通过 `source_record_type + source_record_id + settlement_type` 做唯一约束，避免重复结算。

#### 4. 前端实现要点

- `Settlements.tsx` 中“按场地结算”与“按项目结算”共用列表组件，只切换查询条件和详情头部指标。
- 列表支持从场地详情页带参跳转，默认筛选当前场地。
- 场地结算详情建议增加“项目分布”和“车辆分布”两个二级 tab，方便对账。

## 4. 跨功能公共约束

### 4.1 审批流

- 变更合同、消纳延期、内拨申请、项目结算、场地结算都建议接入统一审批流。
- 审批规则复用 `sys_approval_actor_rule`，推荐固定以下 `biz_type`：
  - `CONTRACT_CHANGE`
  - `CONTRACT_EXTENSION`
  - `CONTRACT_TRANSFER`
  - `PROJECT_SETTLEMENT`
  - `SITE_SETTLEMENT`

### 4.2 数据权限

- 所有合同、入账、结算查询必须带 `tenant_id`。
- 项目、场地、单位三类查询建议同时预留：
  - `(tenant_id, project_id, status)`
  - `(tenant_id, site_id, status)`
  - `(tenant_id, construction_org_id, sign_date)`
- 后端查询层直接做数据权限过滤，不依赖前端隐藏。

### 4.3 与其他模块的依赖

- `项目结算 / 场地结算` 依赖后续 `消纳清单/消纳记录` 真实化。
- `单位统计` 依赖组织、单位主数据完整。
- 附件上传若暂未接对象存储，可先落本地文件表，接口保持不变。

## 5. 验收标准

- 合同列表、合同详情、合同入账、结算管理、月报统计均切到真实接口。
- 合同导入、导出形成闭环，错误数据可回溯。
- 变更、延期、内拨、项目结算、场地结算均有草稿、提交、审批、回写四段状态。
- 月报统计和单位统计支持按月筛选并导出。
