# 00. 多租户用户体系需求文档

> 文档版本：v1.0
> 日期：2026-03-31
> 项目：数智生态渣土运营管理系统

---

## 一、模块概述

### 1.1 业务背景

系统面向桐庐县渣土运营管理场景，涉及多类参与主体：

| 角色类型 | 说明 |
|----------|------|
| 平台运营方 | 杭州畅桐信息科技有限公司，负责整体平台运营与管理 |
| 监管机构 | 桐庐县交通发展投资集团，负责行政监管与执法 |
| 施工/建设单位 | 出土方，负责出土打卡、合同申请 |
| 运输单位 | 车队/司机，负责渣土运输 |
| 消纳场地管理方 | 场地运营方，负责进场消纳确认 |
| 执法中队 | 负责违规处置与执法核查 |

多租户架构需支持以上主体在统一平台内**数据隔离、权限独立、角色自定义**运营。

### 1.2 设计目标

- 支持 SaaS 多租户模型，不同租户数据严格隔离
- 租户内支持多级组织架构（最多 4 级）
- 权限粒度到菜单 + 按钮级别
- 数据权限按组织/场地/项目维度过滤
- 审批流程按租户独立配置

## 三、详细需求

### 2.1 租户管理

#### 需求理解
平台运营方（超级管理员）对入驻租户进行生命周期管理，每个租户代表一个独立的运营主体（如某县市的渣土管理运营单位）。

#### 业务分析
- 租户是数据隔离的最顶层单位
- 每个租户拥有独立的组织树、用户、角色、配置
- 租户间数据完全隔离，跨租户访问需显式授权

#### 功能边界
| 功能项 | 说明 |
|--------|------|
| 租户列表 | 展示所有租户，��持名称、状态筛选 |
| 新增租户 | 创建租户基本信息，分配超管账号 |
| 编辑租户 | 修改租户名称、联系人、有效期 |
| 禁用/启用租户 | 禁用后该租户所有用户无法登录 |
| 租户配置 | 配置租户专属参数（LOGO、短信签名、地图范围等） |

#### 字段说明
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 租户编码 | string | 是 | 全局唯一，系统自动生成 |
| 租户名称 | string | 是 | 最大50字 |
| 联系人 | string | 是 | - |
| 联系电话 | string | 是 | 手机号格式 |
| 有效期 | date | 否 | 为空则永久有效 |
| 状态 | enum | 是 | 启用/禁用 |
| 备注 | string | 否 | 最大200字 |

#### 按钮权限
- `tenant:list` 查看列表
- `tenant:create` 新增租户
- `tenant:edit` 编辑
- `tenant:disable` 禁用/启用
- `tenant:config` 租户配置

#### 接口描述
```
GET  /api/tenants              查询租户列表（分页）
POST /api/tenants              创建租户
PUT  /api/tenants/{id}         更新租户
PUT  /api/tenants/{id}/status  切换状态
GET  /api/tenants/{id}/config  获取配置
PUT  /api/tenants/{id}/config  更新配置
```

### 2.2 组织管理

#### 需求理解
租户内部按组织架构管理人员，支持公司-部门-小组多级树形结构，同时区分「执法组织」和「公司单位组织」两种类型。

#### 业务分析
- 组织是人员归属的载体，也是数据权限的基础单元
- 执法组织（如执法中队）需要额外配置执法权限，可查看全辖区违规数据
- 公司单位组织（建设/施工/运输/场地）仅查看与自身相关数据

#### 功能边界
| 功能项 | 说明 |
|--------|------|
| 组织树展示 | 树形结构展示，支持折叠/展开 |
| 新增组织 | 在任意节点下挂载子组织 |
| 编辑组织 | 修改名称、类型、负责人 |
| 删除组织 | 软删除，有子节点或绑定人员时禁止删除 |
| 搜索组织 | 关键词搜索，高亮匹配节点 |
| 组织类型 | 区分「执法组织」和「单位组织」，独立管理 |

#### 字段说明
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 组织名称 | string | 是 | 同级不允许重名 |
| 组织类型 | enum | 是 | 执法组织 / 单位组织 |
| 上级组织 | ref | 否 | 为空则为顶级 |
| 负责人 | ref(User) | 否 | 关联系统用户 |
| 排序 | int | 否 | 控制展示顺序 |
| 备注 | string | 否 | - |

#### 内容限制
- 组织树最大深度：4 级
- 单租户最大组织数：500
- 组织名称：最长 50 字

#### 按钮权限
- `org:list` 查看
- `org:create` 新增
- `org:edit` 编辑
- `org:delete` 删除

#### 接口描述
```
GET  /api/orgs/tree            获取组织树（含子节点）
GET  /api/orgs                 组织列表（分页）
POST /api/orgs                 创建组织
PUT  /api/orgs/{id}            更新组织
DELETE /api/orgs/{id}          删除组织
GET  /api/orgs/{id}/members    查看组织下人员
```

### 2.3 人员管理

#### 需求理解
管理系统内所有登录用户，包括平台内部人员、施工方人员、场地管理员等，控制其登录权限与数据可见范围。

#### 业务分析
- 人员绑定组织决定数据权限归属
- 人员可配置多个角色，权限取并集
- 人员与小程序账号关联（手机号作为统一标识）
- 支持人员的启用/禁用控制登录能力

#### 功能边界

**人员列表**
- 展示：姓名、手机号、所属组织、角色、状态、最后登录时间
- 筛选：姓名/手机号、组织、角色、状态
- 支持导出 Excel

**人员新增**
- 填写基础信息（姓名、手机号、邮箱）
- 选择所属组织（单选）
- 分配角色（多选）
- 设置初始密码或发送短信激活

**人员详情**
- 查看完整信息
- 修改角色/组织
- 重置密码
- 禁用/启用

#### 字段说明
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| 姓名 | string | 是 | 最长20字 |
| 手机号 | string | 是 | 唯一标识，格式校验 |
| 邮箱 | string | 否 | 格式校验 |
| 所属组织 | ref(Org) | 是 | 单选 |
| 角色 | ref(Role)[] | 是 | 至少1个 |
| 状态 | enum | 是 | 启用/禁用 |
| 备注 | string | 否 | - |

#### 按钮权限
- `user:list` 查看列表
- `user:create` 新增
- `user:edit` 编辑
- `user:delete` 删除
- `user:reset-pwd` 重置密码
- `user:disable` 禁用/启用
- `user:export` 导出

#### 接口描述
```
GET  /api/users              查询用户列表（分页+筛选）
POST /api/users              创建用户
GET  /api/users/{id}         查询用户详情
PUT  /api/users/{id}         更新用户
PUT  /api/users/{id}/status  启用/禁用
PUT  /api/users/{id}/reset-password  重置密码
GET  /api/users/export       导出用户列表
```

### 2.4 角色管理

#### 需求理解
角色是权限的集合，管理员可以自定义角色并为其分配菜单权限和按钮权限，再将角色分配给人员。

#### 业务分析
- 系统预置若干内置角色（超级管理员、租户管理员、场地管理员、执法人员等）
- 内置角色不允许删除，可以修改权限
- 自定义角色可创建/编辑/删除
- 菜单权限和按钮权限分开配置，按钮权限依赖父菜单权限

#### 功能边界

**角色列表**
- 展示角色名称、描述、人员数量、创建时间
- 区分系统内置角色（标记）和自定义角色

**角色详情/编辑**
- 树形展示所有菜单，勾选可访问菜单
- 菜单节点展开后显示该菜单下的按钮权限（细粒度）
- 保存后立即生效（在线用户下次操作时刷新）

**角色新增**
- 输入名称、描述
- 选择菜单权限（树形勾选）
- 选择按钮权限

#### 预置角色说明
| 角色 | 权限范围 |
|------|----------|
| 超级管理员 | 全部权限（仅平台方使用） |
| 租户管理员 | 租户内所有功能 |
| 合同管理员 | 合同结算相关功能 |
| 项目管理员 | 项目管理相关功能 |
| 场地管理员 | 消纳场地管理功能 |
| 执法人员 | 违规车辆、执法信息查询 |
| 统计查询员 | 只读统计报表 |
| 小程序用户 | 移动端功能 |

#### 按钮权限
- `role:list` 查看
- `role:create` 新增
- `role:edit` 编辑权限
- `role:delete` 删除
- `role:assign` 分配人员

#### 接口描述
```
GET  /api/roles                  查询角色列表
POST /api/roles                  创建角色
GET  /api/roles/{id}             角色详情（含权限树）
PUT  /api/roles/{id}             更新角色
DELETE /api/roles/{id}           删除角色
PUT  /api/roles/{id}/permissions 更新角色权限
GET  /api/permissions/tree       获取权限树
```

### 2.5 权限管理

#### 需求理解
权限分为两个维度：菜单权限（控制能看到哪些页面）和按钮权限（控制页面内能做哪些操作）。

#### 权限模型
```
Permission（权限点）
├── type: MENU     菜单权限
│   └── 对应前端路由，控制菜单显示
└── type: BUTTON   按钮权限
    └── 对应前端按钮/操作，控制操作入口显示
```

#### 权限编码规范
```
格式：{模块}:{资源}:{操作}
示例：
  contract:list:view    合同列表查看
  contract:receipt:create  合同入账创建
  site:list:export      场地列表导出
  vehicle:violation:disable  车辆禁用操作
```

#### 功能边界
- 权限点由开发人员在代码中定义，系统启动时自动注册
- 管理员不可自行新增权限点，只能分配
- 权限点与菜单关联，删除菜单时对应按钮权限同步失效

#### 接口描述
```
GET /api/me/permissions   获取当前用户权限列表（登录时调用）
GET /api/menus/tree       获取当前用户可见菜单树
```

### 2.6 数据权限（数据范围）

#### 需求理解
控制用户能查看哪些数据，与功能权限正交。例如：同���是「查看合同列表」的功能，A 用户只能看自己组织的合同，B 用户可以看全部。

#### 数据范围类型
| 类型 | 说明 |
|------|------|
| 全部数据 | 不限制，查看所有数据（超管/租户管理员） |
| 本组织及下级 | 查看本组织及子组织产生的数据 |
| 本组织 | 只看本组织数据，不包含子组织 |
| 本人数据 | 只看本人操作的数据 |
| 自定义范围 | 手动指定可访问的组织/场地/项目列表 |

#### 业务分析
- 数据范围绑定在角色上，随角色分配给用户
- 涉及数据范围过滤的模块：合同、项目、场地、车辆、预警、事件
- 执法人员需要跨单位查看特定区域数据，使用「自定义范围」

#### 实现要点
- 后端查询时自动注入数据范围过滤条件（AOP拦截）
- 前端无需感知，由后端透明处理
- 日志记录数据范围变更

#### 接口描述
```
GET  /api/data-scopes          查询数据权限配置
PUT  /api/data-scopes/{roleId} 更新角色数据范围
```

### 2.7 审核审批配置

#### 需求理解
平台内多类业务流程需要审核审批（合同申请、变更申请、延期申请、事件上报等），审批流程支持租户自定义配置。

#### 业务分析
- 审批事项类型：合同申请、合同变更、合同延期、内拨申请、事件上报、维修申请等
- 每类事项支持独立配置审批节点和审批人
- 支持串行审批（逐级）
- 支持审批退回并填写退回原因
- 审批结果通过消息通知申请人

#### 功能边界

**配置列表**
- 展示所有审批事项的配置状态
- 支持启用/禁用某类审批流程

**流程配置**
- 为每类事项配置审批节点（最多5级）
- 每级审批人：指定人员 或 指定角色
- 审批方式：任意一人通过 / 全部通过

**办事材料配置**
- 为每类事项配置提交材料清单
- 每项材料设置：名称、是否必填、文件类型限制、是否启用

#### 字段说明（办事材料）
| 字段 | 类型 | 说明 |
|------|------|------|
| 材料名称 | string | 如「营业执照」 |
| 是否必填 | boolean | 控制申请时是否强制上传 |
| 文件类型 | string[] | 如 [pdf, jpg, png] |
| 是否启用 | boolean | 关闭后申请流程中不展示 |

#### 按钮权限
- `approval:config:view` 查看配置
- `approval:config:edit` 编辑流程
- `approval:material:edit` 编辑材料配置

#### 接口描述
```
GET  /api/approval-configs                    查询审批配置列表
GET  /api/approval-configs/{type}             查询指定类型的审批配置
PUT  /api/approval-configs/{type}             更新审批流程
GET  /api/approval-material-configs           查询材料配置
PUT  /api/approval-material-configs/{type}    更新材料配置
GET  /api/approval-actor-rules/{configId}     查询审批人配置
PUT  /api/approval-actor-rules/{configId}     更新审批人配置
```

### 2.8 系统日志

#### 需求理解
记录系统运行中的关键事件，为安全审计、问题追查提供依据。

#### 功能边界

**登录日志**
- 记录：用户名、登录时间、IP地址、设备类型、登录结果（成功/失败）
- 筛选：用户名、时间范围、登录结果
- 支持导出

**操作日志**
- 记录：操作人、操作时间、操作模块、操作内容（新增/修改/删除）、操作前后数据对比
- 筛选：操作人、时间范围、模块、操作类型
- 支持导出

**错误日志**
- 记录：发生时间、错误级别、错误信息、堆栈信息、请求URL
- 筛选：时间范围、错误级别
- 支持查看详情

#### 内容限制
- 日志保留周期：登录日志 180 天，操作日志 365 天，错误日志 90 天
- 不记录查询类操作（仅记录写操作）
- 敏感字段（密码等）自动脱敏

#### 按钮权限
- `log:login:view` 查看登录日志
- `log:operation:view` 查看操作日志
- `log:error:view` 查看错误日志
- `log:export` 导出日志

#### 接口描述
```
GET /api/login-logs      查询登录日志（分页+筛选）
GET /api/operation-logs  查询操作日志（分页+筛选）
GET /api/error-logs      查询错误日志（分页+筛选）
GET /api/*/export        导出各类日志
```

## 四、业务流程

### 4.1 用户登录流程

```
用户输入账号密码
    → 后端验证账号/密码
    → 验证租户状态（是否禁用/过期）
    → 验证用户状态（是否禁用）
    → 生成 JWT Token（含租户ID、用户ID、权限列表）
    → 返回 Token + 菜单树 + 用户信息
    → 前端缓存权限，渲染菜单
```

### 4.2 权限鉴权流程

```
前端路由守卫：
    访问页面 → 检查 Token → 检查菜单权限 → 放行/跳转403

后端接口鉴权：
    请求进入 → JWT 解析 → 权限注解校验 → 数据范围注入 → 业务处理
```

### 4.3 新增用户流程

```
管理员操作新增
    → 填写基础信息 + 选择组织 + 分配角色
    → 系统检查手机号唯一性
    → 创建用户（初始密码 = 手机号后6位 或 发短信激活）
    → 记录操作日志
    → 通知用户（短信）
```

### 4.4 审批流程

```
用户提交申请（填写内容 + 上传材料）
    → 系统匹配审批配置
    → 生成审批任务推送给第一级审批人
    → 审批人通过 → 流转下一级 / 审批人退回 → 退回原因通知申请人
    → 所有级别审批通过 → 业务流转执行
    → 全程消息通知
```

## 五、技术方案要点

### 5.1 多租户隔离方案

```
方案：行级租户隔离（Shared Database, Shared Schema）

实现：
- 所有业务表增加 tenant_id 字段
- 通过 MyBatis 拦截器自动注入 tenant_id = 当前登录用户租户
- 查询条件自动追加 WHERE tenant_id = ?
- 超管账号 tenant_id = null，不受限制
```

### 5.2 JWT Token 结构

```json
{
  "sub": "userId",
  "tenantId": "tenant001",
  "orgId": "org-xxx",
  "roles": ["role_contract", "role_site"],
  "permissions": ["contract:list:view", "site:list:view"],
  "dataScope": "ORG_AND_CHILDREN",
  "exp": 1234567890
}
```

### 5.3 权限缓存

- Redis 缓存用户权限点列表，Key = `perm:user:{userId}`
- TTL = 30 分钟，修改角色后主动清除缓存
- 前端本地存储菜单树，路由守卫读取

### 5.4 数据权限注入

```java
// AOP 切面，在 Service 层方法执行前注入数据范围条件
@DataScope(orgAlias = "org", userAlias = "user")
public PageResult<Contract> listContracts(ContractQuery query) { ... }
```

## 七、移动端（小程序）用户体系

### 7.1 登录方式
- 账号密码 + 短信验证码（二次校验）
- 微信扫码绑定（绑定后支持微信免密登录）

### 7.2 权限控制
- 小程序权限与 PC 端权限共用同一用户体系
- 小程序菜单独立配置（部分功能仅在移动端展示）
- 场地人员登录后仅看到消纳相关功能
- 施工方人员登录后仅看到出土打卡相关功能

### 7.3 账号绑定流程
```
用户在小程序登录（手机号+密码）
    → 验证成功后提示绑定微信
    → 调用微信授权，获取 openId
    → 将 openId 与系统用户绑定
    → 后续通过微信登录直接关联用户，免密进入
```

### 7.4 相关接口
```
POST /api/mini/login              手机号+密码登录
POST /api/mini/login/sms          短信验证码登录
POST /api/mini/bind-wechat        绑定微信
GET  /api/mini/me                 获取当前用户信息
PUT  /api/mini/me/password        修改密码
POST /api/mini/feedback           提交问题反馈
```
