## 一、小程序登录两次失败说明

**现象**：  
调用 `POST /api/mini/auth/login`（账号 driver1 / testuser1）时，接口返回 **401，「未登录或 token 缺失」**。

**原因**：  
请求会先经过 **JwtAuthFilter**。当前 8090 上运行的进程里，对「小程序登录」路径的放行条件未覆盖实际收到的请求路径（例如网关/代理或 Servlet 路径差异导致 `request.getRequestURI()` / `getServletPath()` 与预期不一致），因此被当成「未带 token 的普通请求」直接返回 401，**未进入 MiniAuthController**。

**已做修复**：  
在 `JwtAuthFilter.shouldNotFilter()` 中已增加更宽松的放行条件：

- `uri.contains("/mini/auth/login") || path.contains("/mini/auth/login")`  
- 保留原有 `uri.startsWith("/api/mini/auth/login")` 等判断  

这样无论请求路径是 `/api/mini/auth/login` 还是被解析为 `/mini/auth/login`，都会放行，不再要求 Bearer token。

**你需要做的**：  
**重启后端**（重启当前占用 8090 的 xngl-service 进程），使新 Filter 生效。重启后再测小程序登录，应返回 200 及 `data.token`、`data.user`（含 `role`、`userType`）。

## 二、小程序能力测试范围（以小程序为主）

以下均围绕**小程序端**能力设计，平台仅作为「账号来源」和「数据支撑」。

| 能力项 | 说明 | 依赖 |
|--------|------|------|
| 小程序登录 | 账号密码登录，返回 token + user（含 role/userType） | 平台先创建账号；后端 Mini 登录接口放行 |
| 登录后跳转 | 司机端账号 → 直接进司机工作台；平台用户 → 进首页（当前单位、项目、快捷入口） | 小程序根据 user.role / userType 判断 |
| 司机工作台 | 今日任务、预警、处置证、打卡异常申报、问题反馈 | 仅司机端账号可见为「首页」 |
| 平台用户首页 | 当前单位、可用项目、出土拍照 / 打卡异常 / 延期 / 反馈 / 司机端 等入口 | 非司机端账号 |
| 司机端边界 | 司机端账号不出现「当前单位」「出土拍照」等平台入口，仅司机工作台内能力 | 首页对司机端重定向到司机工作台 |

## 五、结论与后续动作

- **两次「失败」**：均为**小程序登录接口被 JWT 过滤器误拦**（返回 401），**不是**账号错误或业务拒绝。  
- **已做**：在 JwtAuthFilter 中增加对 `/mini/auth/login` 的放行；重启后端后 Mini 登录应返回 200。  
- **建议**：  
  1. **重启后端**，再测 `POST /api/mini/auth/login`（testuser1、driver1 各一次），确认 200 及 `data.user.role` / `data.user.userType`。  
  2. 在**小程序开发工具/真机**用上述账号登录，验证：  
     - 平台用户 → 首页；  
     - 司机端 → 直接司机工作台，且再次打开仍为司机工作台。  
  3. 在司机工作台内逐项点「今日任务」「预警」「处置证」「打卡异常申报」「问题反馈」，做一次联调验收。

完成上述步骤后，将「实际结果」与「状态」填回本报告第三节表格，即得到以**小程序能力**为主的完整测试报告。
