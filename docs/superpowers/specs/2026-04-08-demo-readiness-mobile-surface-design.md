# 正式演示达标包设计

## 1. 背景
面向甲方正式评估型演示视频，当前 PC 管理端的大部分核心模块已经具备真实页面与后端闭环，真正影响“是否被判断为已实现”的风险集中在三类：

1. 移动端/小程序缺少可运行的正式演示入口。
2. 处置证同步后的业务闭环没有在项目详情页真实体现。
3. 平台对接页面仍保留 `mock` 语义，不适合直接对外演示。

本次设计目标不是扩大功能面，而是把现有能力收束成可录制、可讲解、可回看的正式演示达标包。

## 2. 目标
本次实现完成后，需要满足以下结果：

1. 系统内新增一个可运行的移动端演示页面，能够真实调用现有 `/api/mini/*` 接口。
2. 项目详情中的处置证区域改为真实数据展示，不再使用前端拼装占位数据。
3. 平台对接和处置证相关页面对外文案去除 `mock` 语义，统一为“手动同步/补拉同步/同步记录”等正式口径。
4. 形成一套稳定的演示账号和演示数据基础，使 10 分钟视频可以按固定路径完成录制。

## 3. 设计选择

### 方案 A：在现有 Web 系统内新增“移动作业端”演示页
优点：

1. 直接复用当前前端工程和部署环境，不需要额外搭建独立小程序前端。
2. 可以用统一账号体系访问现有 mini 接口，方便联调和录屏。
3. 能在同一系统中快速形成“移动端提交 -> PC 端回看”的完整闭环。

限制：

1. 形态是“移动端演示壳”，不是独立微信小程序发布包。
2. 需要在页面内自管 mini token，不能直接复用 PC 登录态。

### 方案 B：继续沿用当前原型图 + 后台回看页
优点：

1. 成本最低。

限制：

1. 无法满足“有真实端侧操作”的演示标准。
2. 甲方容易判断为方案展示而非系统实现。

本次采用方案 A。

## 4. 功能边界

### 4.1 本次必须覆盖
1. 移动端登录、短信验证码、账号绑定。
2. 出土拍照或现场打卡样例提交。
3. 人工消纳/消纳确认样例提交。
4. 事件上报或问题反馈样例提交。
5. 车辆实时跟踪与轨迹查看。
6. 项目详情真实处置证展示。
7. 平台对接正式化文案收口。

### 4.2 本次不扩大范围
1. 不新建独立微信小程序工程。
2. 不重做后端 mini 领域能力。
3. 不新增复杂审批流逻辑。
4. 不处理深层生产化能力，如弱网缓存、消息推送、离线同步。

## 5. 页面与交互设计

### 5.1 移动作业端页面
新增一个单页式“移动作业端”演示页面，采用手机卡片式布局，包含以下分区：

1. `登录区`：账号密码、手机号、验证码、登录、openId 绑定、openId 登录。
2. `现场作业区`：出土拍照、人工消纳、事件上报、问题反馈。
3. `车辆监管区`：实时车辆、轨迹、停留点。
4. `我的信息区`：当前用户、可访问场地、绑定状态。

页面以“真实提交 + 列表回看”的方式组织，不只展示表单。

### 5.2 项目详情处置证区块
将当前 mock 列表替换为真实处置证数据：

1. 按项目 ID 拉取处置证列表。
2. 展示证号、证件类型、绑定状态、车辆、场地、有效期。
3. 提供跳转到处置证清单的快捷入口。

### 5.3 平台对接页文案
仅调整对外展示口径，不改变底层接口路径：

1. `mock 同步` 改为 `手动同步`。
2. `mock 数据已同步` 改为 `同步记录已生成`。
3. 将说明文案统一改成“补拉、同步、状态、映射、控制、票据”等正式表述。

## 6. 数据流

### 6.1 移动作业端
1. 用户在演示页输入账号、密码、手机号，调用 `/api/mini/auth/send-sms-code` 获取验证码。
2. 使用验证码调用 `/api/mini/auth/login`，演示页内部保存 mini token。
3. 后续所有 mini 请求都使用该 mini token 单独鉴权。
4. 现场提交后立即刷新对应列表，形成“提交 -> 留痕回看”闭环。

### 6.2 处置证闭环
1. 在平台对接页发起政务同步。
2. 同步结果写入处置证台账与同步批次。
3. 项目详情通过真实处置证接口读取与该项目关联的数据。

## 7. 验证口径
完成后至少验证以下内容：

1. PC 端构建通过。
2. 移动作业端可登录并提交至少一条真实记录。
3. 项目详情不再出现“未沉淀真实处置证数据”占位文案。
4. 平台对接页面不再出现对外可见 `mock` 文案。
5. 移动作业端提交的数据能在列表或后台页中回看。
