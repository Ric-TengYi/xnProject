## 测试范围说明（逐项回答）

| 问题 | 结论 | 说明 |
|------|------|------|
| **实际测了哪些？** | 登录、/me、/me/menus、用户列表/详情/状态、角色列表、角色权限查询 GET、组织树、菜单树 GET | 未做：角色权限保存后的登录+菜单加载联调、子用户登录+权限校验、菜单管理 CRUD、按钮权限的 UI 编辑与接口联调 |
| **角色的菜单权限编辑** | 接口已测（GET /roles/:id/permissions）；保存（PUT）未在自动化中覆盖 | 前端角色页有菜单树勾选 + 保存配置（PUT menuIds + permissionIds），需浏览器点选保存后刷新验证 |
| **按钮权限编辑** | 后端支持（PUT /roles/:id/permissions 接收 permissionIds）；前端无单独按钮权限勾选 UI | 角色页仅展示菜单树；permissionIds 从接口带回并原样回传，没有「按按钮维度勾选」的界面 |
| **绑定用户后是否做了账号登录+菜单加载测试** | **未做** | 应补充：给某用户分配仅部分菜单的角色 → 该用户登录 → 调 /me/menus 与 /me/permissions，校验仅返回该角色范围内的菜单与按钮 |
| **角色编辑后，角色对应用户是否更新** | **实时生效，无需单独“更新用户”** | 权限按角色实时计算：/me/menus、/me/permissions 每次请求按当前用户的角色重新查 role_menu_rel、role_permission_rel，故改角色后用户下次请求即得到新菜单/权限 |
| **子用户登录后，最大权限是否为自己所拥有的权限** | **未做子用户登录+权限校验** | 需用仅赋权部分菜单/按钮的子账号登录，调 /me/menus、/me/permissions，确认返回集合 ⊆ 该角色配置，且无越权菜单/按钮 |
| **菜单表是否维护了按钮的权限** | **是，通过权限表关联** | sys_menu 有 permission_code（单码）；按钮级权限在 sys_permission，通过 menu_id 关联菜单，resource_type 为 BUTTON/ACTION；角色通过 sys_role_permission_rel 关联 permission_id |
| **是否有菜单管理模块** | **后端有，前端无** | MenusController 提供 GET /tree、GET /:id、POST、PUT /:id、DELETE /:id；前端无「菜单管理」路由与页面 |
| **菜单管理模块是否有 CRUD 测试** | **未做** | 未对 POST/PUT/DELETE /api/menus 及 GET /api/menus/:id 做用例与执行 |

## 一、本轮开发完成项

## 六、测试缺口与补充测试

### 6.1 缺口汇总

- **角色菜单权限编辑**：仅测了 GET /roles/:id/permissions，未测 PUT 保存后再次 GET 及登录后 /me/menus 是否一致。
- **按钮权限编辑**：后端 PUT 支持 permissionIds；前端无按钮维度勾选 UI，未做「改 permissionIds 后子用户 /me/permissions」联调。
- **绑定用户后的登录+菜单加载**：未用「仅绑定了某角色」的账号登录并校验 /me/menus、/me/permissions。
- **角色编辑对应用户是否更新**：逻辑为实时按角色计算，无需额外测试「更新用户」；可测「改角色权限 → 用该角色用户登录 → 看菜单/权限是否变化」。
- **子用户最大权限**：未用子账号登录并校验返回的菜单/按钮 ⊆ 其角色配置。
- **菜单表与按钮权限**：已确认 sys_permission.menu_id 关联菜单；未测权限列表按 menuId 过滤等。
- **菜单管理模块**：后端 CRUD 存在；前端无页面；未对 POST/PUT/DELETE /api/menus 及 GET /api/menus/:id 做用例执行。

## 七、结论与后续

- **开发**：组织与人员编辑/停用、角色菜单与 permissionIds 及半选保存、侧栏 /me/menus 动态菜单与兜底、010 菜单种子均已实现。
- **测试**：登录、/me、/me/menus、用户列表与详情与状态、角色列表与权限、组织树、菜单树等接口均通过；**未覆盖**：角色权限 PUT 后联调、子用户登录+菜单/权限、菜单管理 CRUD、按钮权限 UI 与接口联调。
- **建议**：
  1. 本地浏览器验证：角色页勾选菜单后保存，刷新看勾选保持；编辑/停用用户。
  2. 准备一个仅绑定「部分菜单+部分按钮」角色的子账号，登录后调 /me/menus、/me/permissions，确认不越权。
  3. 若需菜单管理：增加前端「菜单管理」页并对接 GET/POST/PUT/DELETE /api/menus，再补 CRUD 与树展示测试。
  4. 若需按钮权限可编辑：在角色页或权限管理页增加「按按钮/接口维度勾选」的 UI，并接 GET/PUT /roles/:id/permissions 的 permissionIds。
